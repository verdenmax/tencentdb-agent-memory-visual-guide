为什么需要 Agent MemoryWhy Agent Memory

TencentDB Agent Memory solves a recurring long-horizon agent problem: the longer an agent works, the more conversation history, tool logs, project rules, and user preferences accumulate, while the context window remains limited. The project does not dump all history back into the prompt; it stores and recalls information by layers.

🔌 Analogy
Think of the context window as a desk. If every document is piled on the desk, the important part disappears. TencentDB Agent Memory behaves like a layered filing cabinet: the desk keeps the current task map, drawers keep summaries, and the archive keeps evidence.

Two memory problems

Long-term memory

Remember preferences, facts, scenes, and persona across conversations, then recall them before the next turn.

Short-term compression

When a long task produces huge tool logs, offload raw text to files and keep a Mermaid task canvas in context.

The two spines

L0Conversation
Raw JSONL conversations and searchable evidence.
L1Atom
Structured facts, preferences, constraints, and events extracted by an LLM.
L2Scene
Readable scene blocks that aggregate atom memories into context.
L3Persona
A stable user profile generated from scenes and injected into system context.

Pseudocode: layer history instead of piling it up

on_turn_committed(messages):
    l0 = record_raw_conversation(messages)
    if pipeline.should_extract_l1():
        atoms = extract_memories(l0)
        store.upsert(atoms)
    if pipeline.should_update_scenes():
        scenes = merge_atoms_into_scene_blocks()
    if pipeline.should_update_persona():
        persona = synthesize_user_profile(scenes)

before_next_turn(user_text):
    memories = search_l1(user_text)
    persona = read_l3_persona()
    return inject(memories, persona)

✅ Key points
The project is not about remembering more text. It preserves structure in upper layers and evidence in lower layers. Long-term memory handles cross-session experience, while context offload handles one long task's log explosion.

两条记忆主线Two memory spines

TencentDB Agent Memory is not one universal memory pipeline. It has two cooperating spines: long-term L0-L3 preserves reusable experience for future sessions, while short-term Offload compresses the current task's tool traces into a navigable, drill-down context map.

🔌 Analogy
Long-term memory is like a team knowledge base: stable rules, preferences, scenes, and persona survive into the next project. Offload is like the whiteboard for this meeting: raw outputs are attached behind the board, while the room sees only the structure needed now.

Two spines, two jobs

Long-term L0-L3

Start with raw conversation, extract atoms, aggregate scenes, synthesize persona, then recall before a later turn or future session. It asks: what experience should be reused later?

Short-term Offload

Start with tool results, write refs markdown and offload JSONL, produce L1/L1.5/L2 Mermaid MMD, inject an L3 task canvas, then drill down by node_id. It asks: how do we reduce current prompt pressure?

The L2 promotion decision is anchored by checkL2Trigger in src/offload/pipelines/l2-mermaid.ts. This spine is not learning the user; it is keeping a long current task manageable.

Shared principle: structure above, evidence below

Upper layers keep structure

L3 persona, L2 scenes, and Offload L2/L3 Mermaid should be compact, stable, and navigable. The model sees the map first.

Lower layers keep evidence

L0 conversations, refs markdown, and offload JSONL preserve raw text, sources, and node_id links so the agent can drill down only when needed.

Why not merge them?

If every Offload trace becomes long-term memory, temporary tool noise pollutes future sessions. If durable preferences live only in Offload, they disappear when the current task ends. Long-term L0-L3 optimizes cross-session reuse: is the fact stable enough to help later decisions? Offload optimizes current-context pressure: how can huge tool outputs become a compact task graph while still preserving evidence for node_id drill-down? They cooperate, but they should not replace each other.

Pseudocode: choose by lifetime, inject by level

if information_should_survive_future_sessions:
    send_to_long_term_memory()
else_if information_is_current_task_trace:
    send_to_context_offload()

when_agent_needs_context():
    inject_high_level_structure()
    drill_down_to_evidence_only_when_needed()

✅ Key points
First ask about lifetime: future sessions go to long-term L0-L3; current-task traces go to Offload. Then ask about injection level: put high-level structure in the prompt, and keep raw evidence below for search, references, or node_id drill-down.

OpenClaw 零配置闭环OpenClaw zero-config loop

This lesson turns L0-L3, Recall, and search into a minimal working OpenClaw loop: install the plugin, enable the smallest config, restart the Gateway, then first verify L0 capture and L1 search/recall. L2/L3 may appear later.

🔌 Analogy
Zero config is like plugging in a default local disk: you can save and read without first designing cloud storage or tuning indexes. For stronger vector recall or larger deployments, you still add real embedding and backend configuration later.

One minimal loop

1

Install or update the plugin

Install @tencentdb-agent-memory/memory-tencentdb with the OpenClaw plugin command; run the matching update command explicitly when upgrading.

openclaw plugins install @tencentdb-agent-memory/memory-tencentdb
openclaw plugins update @tencentdb-agent-memory/memory-tencentdb

2

Enable the smallest config

Set only enabled in ~/.openclaw/openclaw.json. Never put real API keys in guide examples.

3

Restart the Gateway

Plugin config, hooks, and tools are registered during the Gateway lifecycle, so a restart makes the new config take effect reliably.

openclaw gateway restart

4

Talk for 2-3 turns

Give the agent a reusable preference or project constraint, then continue the conversation so capture first lands in L0 raw conversation and L1 structured memory.

5

Verify files, recall, and search

First check for artifacts under ~/.openclaw/state/memory-tdai/, such as conversations/, records/, scene_blocks/, or vectors.db; then verify L0 capture and L1 search/recall. L2 scene blocks and L3 persona may appear only after pipeline delay or trigger conditions.

The minimal config

{
  "memory-tencentdb": {
    "enabled": true
  }
}

Zero config means the plugin can use the default local SQLite plus sqlite-vec path for course work, personal trials, and offline smoke tests. It does not mean every capability is automatically optimal: when no usable embedding is configured, vector behavior may be disabled or degraded by default. Start with the loop, then tune embeddings, remote storage, retention, and observability only when the loop is proven.

Minimal daily config vs advanced tuning

Minimal daily config

Use only enabled: true with the default local SQLite + sqlite-vec path. Verify plugin registration, L0 raw capture, and L1 search/recall first; L2 scene blocks and L3 persona may appear later after pipeline delay or trigger conditions.

Advanced tuning

Add embeddings, remote databases, TCVDB, retention policy, debug logging, and performance settings as needed. Real API keys belong in secure local config or environment variables, not in public examples.

Pseudocode

openclaw plugins install @tencentdb-agent-memory/memory-tencentdb
openclaw plugins update @tencentdb-agent-memory/memory-tencentdb

# ~/.openclaw/openclaw.json
{
  "memory-tencentdb": { "enabled": true }
}

openclaw gateway restart

Smoke test: observe three things

  • Check file artifacts first: inspect ~/.openclaw/state/memory-tdai/; expected local state artifacts include conversations/, records/, scene_blocks/, and vectors.db.
  • Verify L0/L1 first: talk for 2-3 turns, state a durable preference or project rule, then ask about it in a later turn or session. First confirm L0 raw conversation capture and L1 structured memory search/recall.
  • Keep tool scope clear: tdai_memory_search searches structured L1 memories; tdai_conversation_search searches raw L0 conversations. L2 scene blocks and L3 persona may appear only after pipeline delay or trigger conditions.
🔒 Credential safety
Guide examples must never contain real API keys, cloud database passwords, or private endpoints. Show placeholders only; use secure local config or environment variables for real runs.

Hermes 与 Gateway 路径Hermes and Gateway path

The Hermes path looks different from OpenClaw: one is a Python Agent memory provider, the other is an OpenClaw plugin and Gateway lifecycle. But it is not a different memory engine. Hermes is an alternate host entry: the provider translates Hermes recall, capture, and search calls into HTTP, sends them to the local Gateway, and the Gateway reaches the same TdaiCore through StandaloneHostAdapter.

🚪 Analogy
OpenClaw and Hermes are two doors into the same library. The front desk differs: OpenClaw uses plugin hooks/tools; Hermes uses the memory_tencentdb provider. Once inside, archiving, searching, recall, and shelf organization are handled by the same core service.

In hermes-plugin/memory/memory_tencentdb/README.md, the provider is described as a thin HTTP client plus process supervisor. It plugs into the Hermes lifecycle, starts or probes the Gateway, maps prefetch to POST /recall, maps sync_turn to POST /capture, and advertises search tool schemas. The real capture, extraction, storage, recall, and pipeline scheduling remain in the Node.js Gateway and core modules.

Two install routes: fresh Docker or existing Hermes

Fresh Docker deployment

The Hermes section in README_CN.md positions Docker as the route for starting a memory-enabled Hermes from scratch. The image bundles hermes-agent and the memory_tencentdb provider, with the Gateway exposed on port 8420. This is best when you want one container path to validate the complete loop.

Install provider into existing Hermes

If Hermes is already installed, link or copy the provider into Hermes's memory provider directory and set memory.provider: memory_tencentdb in ~/.hermes/config.yaml. The directory name must be exactly memory_tencentdb because Hermes uses it as the provider key.

Whether you enter through Docker or an existing Hermes install, the checkpoints are the same: Hermes discovers the provider, the provider reaches the local Gateway, and the Gateway HTTP routes reach TdaiCore. That is why the guide does not treat Hermes as a separate memory system: it changes host integration and process boundaries, not the L0-L3, Recall/Search, or pipeline core.

Pseudocode

memory:
  provider: memory_tencentdb

MEMORY_TENCENTDB_GATEWAY_HOST="127.0.0.1"
MEMORY_TENCENTDB_GATEWAY_PORT="8420"

curl http://127.0.0.1:8420/health

The example uses placeholders and a loopback address only. Real LLM credentials, Gateway API keys, cloud database passwords, and private team endpoints do not belong in guide examples, screenshots, or commits. If authentication is enabled, the Gateway side uses TDAI_GATEWAY_API_KEY or server.apiKey; the Hermes provider, as the client, can use MEMORY_TENCENTDB_GATEWAY_API_KEY. They must match, but this lesson never shows a real value.

What the Gateway exposes

src/gateway/server.ts marks the Gateway as the HTTP server for the Hermes sidecar: GET /health is the health check, POST /recall performs memory recall before the model turn, POST /capture captures a conversation turn, POST /search/memories searches L1 structured memories, and POST /search/conversations searches L0 raw conversations. src/gateway/config.ts also exposes optional knobs such as server.apiKey / TDAI_GATEWAY_API_KEY and server.corsOrigins / TDAI_CORS_ORIGINS. The default learning path is a local sidecar; auth and CORS become critical when you expose it beyond loopback.

运行后会出现哪些文件Runtime files and data products

Runtime files are not hidden implementation details. They are a debugging map: when recall, persona, scenes, or long-task summaries look wrong, start with the high-level artifact to see what the system believes, then drill down to the evidence.

🗺️ Analogy
Think of the files as a detective case folder: persona and scenes are summaries, L1 atoms are evidence cards, L0 conversations are recordings; Offload MMDs are task boards, with JSONL and refs as the source bags behind them.

Where L0-L3 land on disk

L0Conversation
conversations/ stores raw conversation JSONL, written by src/core/conversation/l0-recorder.ts, and preserves bottom-level evidence.
L1Atom
records/ stores structured memory JSONL, written by src/core/record/l1-writer.ts, for search and recall.
L2Scene
scene_blocks/ stores Markdown scene blocks, produced by src/core/scene/scene-extractor.ts from multiple L1 records.
L3Persona
persona.md stores long-term persona, generated by src/core/persona/persona-generator.ts from scenes.

The long-term memory directory usually also contains vectors.db, the local SQLite vector/full-text search store; use src/core/store/factory.ts to see how storage is created. .metadata/ keeps manifest and checkpoint clues, connected to src/utils/manifest.ts and src/utils/checkpoint.ts; .backup/ keeps backups for rollback before upgrades or debugging.

Long-term memory vs Offload directories

Long-term memory: cross-session reuse

memory-tdai/ lives under the resolved OpenClaw state dir, commonly ~/.openclaw/state/memory-tdai/; standalone or fallback paths may differ. Its conversations/, records/, scene_blocks/, persona.md, vectors.db, .metadata/, and .backup/ answer whether a future session can remember, search, and explain something. The evidence chain is L3 -> L2 -> L1 -> L0.

Context Offload: current-task compression

Under ~/.openclaw/context-offload/<agent>/, expect refs/, mmds/, offload-<session>.jsonl, state.json, and sessions-registry.json. It answers how the current long task moved large evidence out of context while keeping it recoverable. See src/offload/storage.ts.

Minimal file shape

memory-tdai/
  conversations/     # L0 raw messages
  records/           # L1 atoms
  scene_blocks/      # L2 scenes
  persona.md         # L3 persona
  vectors.db         # local search index

context-offload/
  <agent>/refs/
  <agent>/mmds/
  <agent>/offload-<session>.jsonl

Offload refs/ keep large source text or referenced material; mmds/ keep Mermaid/MMD task diagrams; offload-<session>.jsonl records offload events for the session. state.json tracks current state, and sessions-registry.json helps find artifacts by session.

Drill-down debugging: structure first, evidence second

1

Persona -> Scene

If the user says "the persona is wrong", open persona.md for the specific claim, then find the related scene_blocks/.

2

Scene -> Atom

A scene keeps structure and narrative, not final proof. Continue to records/ and inspect the L1 atom extraction.

3

Atom -> Conversation

If the atom is suspicious, return to conversations/ L0 JSONL and verify the raw user words, time, and context.

4

MMD node -> offload JSONL -> refs

If a long-task node is wrong, inspect the mmds/ node, then offload-<session>.jsonl, then the raw material in refs/.

配置层级与安全默认值Configuration levels and safe defaults

More configuration does not mean a better setup. The safe mental model is progressive: make the zero-config loop run, tune daily knobs only when needed, then move into long-session, backend, LLM, and operational settings.

🎚️ Analogy
Configuration is like a mixing console: everyday use changes volume and mute; long sessions tune compression and EQ; remote devices require drivers, endpoints, and credentials. Defaults should let beginners hear sound before they touch dangerous knobs.

Three configuration levels

1Daily knobs
Tune capture, recall, and a little pipeline: capture on/off, recall strategy, result count, timeout, and minimal trigger thresholds.
2Long-session tuning
For long sessions, tune extraction, pipeline, offload, and bm25: extraction cadence, batching, keyword search, context offload, and reports.
3Backend / LLM / ops
Only then configure embedding, tcvdb, llm, report, and gateway authentication, because these involve remote services, model dimensions, API keys, backend choice, and operations.

The levels are a learning order, not a permission model. parseConfig combines user settings with defaults into runtime config, so documentation should encourage small, explainable, reversible changes.

Key config groups

  • capture: whether conversations enter L0/L1, plus retention such as l0l1RetentionDays; 0 means cleanup is disabled.
  • extraction: how raw conversations become structured memories after capture is proven stable.
  • pipeline: PipelineTriggerConfig decides when L1, scenes, persona, or batch work runs.
  • recall: RecallConfig manages strategy, result count, and timeoutMs; the timeout avoids blocking the user.
  • embedding: EmbeddingConfig selects local or remote vector behavior; remote providers require apiKey, baseUrl, model, and dimensions.
  • bm25: keyword retrieval for terms, file names, error codes, and project-specific names.
  • tcvdb: Tencent Cloud VectorDB backend settings; the backend switches to TCVDB only when top-level storeBackend: "tcvdb" is also set.
  • offload: OffloadConfig controls current-task compression, refs, MMDs, and session state.
  • llm: model calls for extraction, summarization, or gateway behavior; real secrets stay in secure local config or environment variables.
  • report: human-readable diagnostics and artifacts, not a replacement for lower-level evidence.

SQLite default

🧩 Analogy

1

2

4

    return performAutoRecall(user_text)

    await ensureSchedulerStarted()
    return performAutoCapture(messages)

Why schedulerStartPromise is a concurrency gate

In the Gateway path, several HTTP requests can enter handleTurnCommitted at the same time. With only a boolean “scheduler started” flag, the first request could flip the flag before the async checkpoint read finishes; later requests would skip startup and update scheduler state too early. When scheduler.start(restoredStates) finally runs, restored old state could overwrite those concurrent updates.

schedulerStartPromise stores the in-flight startup itself. Every concurrent request awaits the same promise; after startup succeeds, the resolved promise becomes a cheap sentinel so later calls do not start the scheduler again.

Why the background task registry exists

performAutoCapture can launch deferred background writes for L0 embeddings. They are fire-and-forget, but shutdown still has to respect them. TdaiCore records those in-flight promises in a background task registry; destroy() waits for them, or a bounded timeout, before closing the vector store and embedding service. That prevents late writes from hitting a closed database connection.

运行时初始化与配置解析Runtime init and config parsing

This lesson connects the plugin shell, adapter, and core to startup order: OpenClaw gives the plugin raw api.pluginConfig; index.ts parses it once for that registration and prepares config, time, and directories first. Then TdaiCore.initialize() starts async store initialization, but does not wait for stores to finish before creating the pipeline manager.

🧩 Analogy
Before a shop opens, the manager reads the roster once, fills default roles, opens the door, and prepares shelves; inventory counting can continue in the background while the register is being wired. Runtime initialization is similar: config and directories become stable first, store initialization proceeds asynchronously, and store-dependent paths later await storeReady or degrade when stores are unavailable.

Startup path: raw config to hooks

1

raw plugin config

index.ts reads api.pluginConfig and logs how many keys arrived.

2

3

4

core starts

core.initialize() confirms dirs again, starts storeReady = initStores() in the background, and creates the pipeline manager synchronously.

5

hooks / tools

index.ts registers hooks/tools after core initialization starts; store-dependent paths await storeReady, and pipeline runners wire after store success or failure.

“Parse once per registration” does not mean “parse once per process.” OpenClaw may call register(api) during plugin scan, Gateway startup, channel bootstrap, or config reload. Each call receives the full config, so the entry parses the current raw object directly instead of reusing stale config.

Why directories come before store readiness

Inside TdaiCore.initialize(), initStores starts as a promise: it maintains the manifest, creates the store bundle, and records degraded mode if initialization fails. At the same time, createPipelineManager can run synchronously because runner wiring is deferred behind storeReady.then(...).catch(...).

So “directories first” is true; “stores are definitely ready before the pipeline manager, hooks, and tools” is not. Auto Capture：对话提交后的捕获Auto Capture after a turn commits

After a turn is committed (agent_end / turn committed), performAutoCapture reliably records the new user and assistant messages into L0. It keeps the capture path short and deterministic: write local L0 first, index it for search when possible, then notify the pipeline scheduler; whether L1 extraction runs is a scheduler decision.

🧩 Analogy
After checkout, the receipt already exists. Auto Capture first saves that receipt in the local ledger so it is not lost, then places its number in a search cabinet, then tells the back-office clerk “new work arrived”. Whether the clerk files it immediately or waits for a batch is scheduling policy, not a decision made at the register.

The post-commit capture path

1

agent_end / turn committed

The host passes the full messages array to performAutoCapture. The model response already exists, so capture no longer changes this turn's generation.

2

checkpoint lock

CheckpointManager.captureAtomically() holds the lock while reading the cursor, calling recordConversation, and advancing the cursor. That prevents concurrent agent_end events from reading the same stale cursor.

3

record L0

recordConversation extracts user / assistant messages for this turn, filters them by position slice and timestamp cursor, sanitizes them, and appends them to conversations JSONL.

4

index

If a vector / FTS store is available, capture writes L0Record metadata into the search index. If it is unavailable, local L0 recording still stands.

5

notify scheduler

Capture calls scheduler.notifyConversation(sessionKey, []). It only says “a new conversation round exists”; scheduler policy decides when L1 runs.

How messages become an L0Record

messages
session history from agent_end
->
filteredMessages
new user / assistant messages, cleaned of recall wrappers and structural noise
->
L0Record
id, sessionKey, sessionId, role, content, recordedAt, timestamp

The important word is “new”. recordConversation prefers the originalUserMessageCount recorded before prompt build and slices the raw messages after that position; if that cache is unavailable, captureAtomically reads the checkpoint-owned afterTimestamp cursor inside the atomic section and passes it to the callback for filtering. The caller does not provide a cursor outside the critical section; checkpoint owns cursor read/update under the same lock. Auto Capture therefore does not keep rewriting the whole history into L0. It appends only the incremental messages that appeared after the committed turn.

Sync path vs deferred embedding path

Sync embedding / VDB path

When the store does not support deferred embedding, capture tries to embed synchronously before upserting L0, then writes the L0Record and vector together. This fits remote or VDB backends that need the vector up front.

Deferred embedding / SQLite path

When the store supports supportsDeferredEmbedding, capture writes metadata + FTS first and returns to the main path quickly. Embedding batch and updateL0Embedding run in the background, avoiding a multi-second embedding call on agent_end.

Pseudocode

performAutoCapture(messages):
    checkpoint.captureAtomically(sessionKey, pluginStartTimestamp, (afterTimestamp) => {
        filtered = recordConversation(messages, afterTimestamp)
        return { maxTimestamp: maxTimestamp(filtered), messageCount: filtered.length }
    })
    ...
    scheduler.notifyConversation(sessionKey, [])

Why capture does not run L1 directly

performAutoCapture is responsible for reliable recording and notification. It should not duplicate policy such as everyNConversations, idle timeout, warm-up, retries, or shutdown flush. MemoryPipelineManager.notifyConversation updates the conversation count, persists state, enqueues L1 when the threshold is reached, and otherwise resets the idle timer. That keeps capture short while scheduling policy stays centralized.

Checkpoint：游标、锁与防重复Checkpoints: cursors, locks, and duplicate prevention

Checkpoints are the duplicate-prevention safety rail in the capture path. Duplicate L0 capture usually does not come from JSONL itself; it comes from concurrent agent_end / Gateway calls, or from a capture that reads a stale cursor. This lesson connects captureAtomically(), position slicing, timestamp cursors, and the scheduler start gate.

🧩 Analogy
Think of L0 as a warehouse intake ledger. If two clerks both see “the last receipt was 100”, both may register the same item as 101. The fix is to lock the ledger: read 100, write the new receipt, advance the cursor to 101, then let the next clerk in.

Without the lock: stale cursors duplicate capture

1

Concurrent entry

OpenClaw agent_end and Gateway capture can submit messages for the same session at nearly the same time.

2

Same old cursor

If the cursor is read outside the critical section, both calls may observe afterTimestamp = T0.

3

Duplicate L0 writes

Both recorders believe messages after T0 are new, so the same user / assistant records are appended twice.

4

Late advance

Writing the checkpoint afterward is too late: duplicate evidence has already entered L0, and L1 may extract it twice.

With the lock: checkpoint file mutations are serialized

1

captureAtomically(sessionKey)

CheckpointManager uses a checkpoint-file async lock to serialize mutations; sessionKey selects the cursor and state owned by that session inside the lock.

2

read checkpoint

The checkpoint file lock first serializes file mutations, then reads/updates the sessionKey-owned cursor and checkpoint state/counters inside the lock; callers do not bring an external stale cursor.

3

record L0

The callback calls recordConversation and writes only messages truly new after the cursor or position slice.

4

advance cursor

If messages were written, the same lock advances the checkpoint to the maximum timestamp from this capture.

Pseudocode

captureAtomically(sessionKey):
    lock(checkpoint_file)
    state = read_checkpoint_file()
    cursor = state.sessions[sessionKey].afterTimestamp
    messages = record_after_cursor(cursor)
    if messages:
        state.sessions[sessionKey].afterTimestamp = max_timestamp(messages)
        write_checkpoint_file(state)
    unlock(checkpoint_file)

Position slice and timestamp cursor solve different problems

position slice

The recorder prefers the cached message count from before prompt build, such as originalUserMessageCount. After commit it slices the array after that position, protecting the turn boundary from timestamp drift, same-millisecond messages, or host timestamp rewrites.

timestamp cursor

When stable raw timestamps exist, checkpoint afterTimestamp can be an incremental-capture fallback and audit clue. If raw timestamps are absent or unreliable, this cursor is weak; missing-cache or restart cases only reduce duplicate risk and provide a weaker guarantee than the turn-boundary protection a position slice would provide.

So they are not substitutes. Position slice protects the structural boundary of “messages added by this turn” and is generally stronger than a timestamp cursor. Timestamp cursor protects the persistent incremental boundary only when stable raw time is available; across missing caches or restarts, it is a weaker fallback rather than an equivalent guarantee.

schedulerStartPromise: restored state must not clobber concurrent capture

->

all callers await one startup promise
->

->
no clobber

TdaiCore.schedulerStartPromise handles a different race: while Gateway restores scheduler state, a simultaneous capture could notify the scheduler, write a fresh count, and then be overwritten by an older restored snapshot. The promise turns startup into a gate, so concurrent callers share the same startup result instead of racing ahead independently.

🧩 Analogy
L0 is the complete recording; L1 is the index card written after the meeting. The card does not copy greetings or repeated sentences. It keeps reusable information in the supported persona, episodic, and instruction categories. Each card still names the recording segments it came from, so later review can return to the evidence.

L1 去重、冲突检测与写入L1 dedup, conflict detection, and write path

L1 extraction turns L0 messages into structured memories, but the extracted batch can contain duplicates or facts that conflict with existing memory. Before writing, batchDedup acts as quality control: recall similar candidates, then decide whether the new memory action is store, update, merge, or skip. It governs searchable L1 facts; it does not delete raw L0 evidence.

🧩 Analogy
L0 is the meeting recording archive; L1 is the team's index-card shelf. You do not cut the recording when a new card is written. Dedup only asks whether the card already exists, should revise an older card, or conflicts with one, so the shelf stays useful.

->

->

batchDedup in src/core/record/l1-dedup.ts runs before batch writes over extracted memories that already have a record_id, returning store/update/merge/skip actions. It prefers available recall for candidates: repeated preferences can be skipped, corrected preferences can update an existing record, mergeable details can be merged, and unrelated new facts can be stored. The decision comes from the LLM prompt in src/core/prompts/l1-dedup.ts.

Why dedup does not discard evidence

L0 evidence layer

Raw conversation JSONL records who said what, when, and under which message ID. It supports audit, replay, and re-extraction; dedup should not rewrite this source evidence.

L1 quality layer

Structured facts are optimized for future recall. Dedup only decides whether a new L1 fact would make the fact store repetitive, stale, or conflicting.

Write path: JSONL plus store upsert

The extraction flow first uses generateMemoryId to preassign a unique record ID. It uses time plus random bytes for the record that may be written; it is not a content-stable hash or the semantic identity chosen by dedup. batchDedup then returns decisions keyed by those IDs, and writeMemory uses the preassigned ID when appending or upserting the L1 record. The store upsert path keeps the same record searchable: local sqlite can provide vector / FTS search, while TCVDB can provide managed vector search. JSONL is durable evidence; the store is the retrieval index. Together they make memory auditable and recallable.

SQLite local store

src/core/store/sqlite.ts fits zero-config or local development. It maintains local vector and FTS indexes so L1 records remain discoverable through similarity or keyword search without a remote service.

TCVDB backend

src/core/store/tcvdb.ts integrates a managed vector database. createStoreBundle wires embedding, store backend selection, and degradation behavior so the writer does not depend on backend details.

1

If vector data exists and the embedding service is available, batchDedup recalls candidates with vector similarity.

2

FTS recall

If vector recall is unavailable but FTS keyword search exists, keyword recall can still provide candidates, with a weaker recall basis.

3

no recall

If neither vector nor FTS recall is available, conflict detection is skipped and the new batch defaults to store.

4

writeMemory applies store / update / merge / skip decisions to JSONL and store writes; operators can manually rebuild the retrieval layer from JSONL when needed.

为什么需要 L2 Scene BlocksWhy L2 scene blocks exist

L1 atom memories are good for searching individual facts, but real activity is usually made of many small facts: a debugging session, a design discussion, or a preference change. L2 Scene Blocks organize scattered L1 atoms into readable, navigable, traceable, growing contexts, then give stable conclusions to L3 Persona.

🗂️ Analogy
L1 is a pile of sticky notes: "prefers Chinese answers", "working on dedup today", "evidence came from message m42". L2 is one page in a project log: it groups notes from the same activity and adds a summary, timeline, heat, and evidence links. L3 is the long-term profile that keeps only traits repeatedly supported by many log pages.

L2 bridges L1 and L3

L1 atoms
searchable fine-grained facts, preferences, events, and constraints
->
L2 scene blocks
activity-level context, timeline, evidence, and summary
->
L3 persona
stable traits and long-term tendencies extracted from scenes
->
recall
inject what is relevant without dumping every atom into the prompt

With only L1, the system can find a single fact, but it struggles to explain which activity the facts belonged to, what happened before or after, which evidence supports the summary, and whether the topic is still hot. If L3 is generated directly from L1, a one-off detail can look like a stable trait. L2 is the middle layer: it preserves event context and activity history so L3 can derive stable persona only from scene-level evidence.

L1 and L2 responsibility boundaries

L1 atom responsibilities

Atomic, searchable, deduplicated. Each record should express one fact and carry type, priority, time, and source_message_ids for recall and evidence drill-down.

L2 scene responsibilities

Human-readable, contextual, navigable. A scene block clusters related L1 records and maintains activity summary, timeline, source references, heat, and update history.

FieldPurpose

updatedWhen new L1 atoms were last merged, forming an activity timeline.
summaryCompact description of the activity context, avoiding injection of every L1 atom.
heatRecent activity or importance signal, so recall and maintenance can prioritize hot scenes.

These are the META keys that parseSceneBlock actually parses and formats. Evidence is not a parsed META field: it stays in the scene narrative, source references, and links that explain which L1 atoms support the summary; when verification is needed, L1 source_message_ids lead back to the raw L0 text.

Together, these fields and narrative references answer the core L2 question: why are these facts related? L2 does not replace L1 search or delete raw L0 text. It organizes relationships, order, and source traceability across many L1 records into a readable block. Recall can inject the relevant scene summary first, then drill through the narrative/source references back to L1 and L0 when verification is needed.

L2/L3 调度与触发规则L2/L3 scheduling and trigger rules

MemoryPipelineManager does not run the full L1/L2/L3 stack after every message. It schedules work by layer: L1 batches by threshold, idle, and shutdown flush; L2 consolidates scenes after successful L1 with delay/min/max/active-window rules; L3 enters a global serial queue after L2 and deduplicates repeated triggers with a pending flag.

⏱️ Analogy
L1 waits until you have "said a small chunk" before taking notes: enough turns trigger immediately, otherwise idle time catches the remainder. L2 is the project review: new notes may pull the review earlier, but continuous chatting must not postpone it forever. L3 is the shared company brief about a person, so only one editor should update it at a time.

The entry point is notifyConversation(sessionKey, messages). It appends messages to the session buffer, increments conversation_count, and refreshes last_active_time. If the count reaches the warm-up or steady-state threshold, it calls enqueueL1; otherwise it resets the L1 idle timer. That batches active chats while still capturing low-frequency chats after they go idle.

L1 resettable timer vs L2 downward-only timer

L1 resettable idle

Every below-threshold conversation calls schedule() again: the old countdown is cancelled and a new countdown starts from now. Its goal is debounce: wait until the user stops, then send the residual buffer to L1.

L2 downward-only

After L1 succeeds, code computes max(now + delay, lastL2 + minInterval) and calls tryAdvanceTo(). If the new time is earlier, it moves earlier; if it is later, the old plan remains, so active sessions cannot keep pushing L2 back.

L2 also has two protections: L2 scheduling paths, including successful runs and retry/skip exits, use maxInterval to arm the next poll so active sessions are periodically consolidated even without fresh L1; when the periodic maxInterval timer fires, sessionActiveWindowHours stops polling cold sessions until the next successful L1 wakes them again. So downward-only does not mean "always sooner"; it balances delay-after-L1 responsiveness, minInterval rate limiting, and the maxInterval guarantee.

Why L3 is globally serialized

L2 organizes scenes per session, but L3 persona summarizes all scenes and writes one high-level profile. If several sessions finish L2 and generate persona concurrently, they may read nearby but inconsistent scene/persona states, and the last writer can overwrite the earlier result. To avoid this global write conflict, enqueueL3 uses one SerialQueue("L3"), one l3Running, and one l3Pending: triggers that arrive during a run only mark pending, then the current L3 run is followed by at most one catch-up run.

How the three strategies differ

keyword

searchByKeyword uses SQLite FTS5. buildFtsQuery turns sanitized words into an FTS query, and BM25 ranks the results. This is good for exact terms, file names, and function names.

embedding

searchByEmbedding embeds the query, then finds L1 memories by cosine similarity. It works when the user asks with different wording, but it depends on embedding resources being ready.

hybrid

Hybrid has two paths. If the store declares nativeHybridSearch, such as TCVDB hybridSearch, the server does dense + sparse/RRF-like retrieval. SQLite and other non-native paths use local searchHybrid(): run keyword and embedding separately, then merge rankings with RRF inside auto-recall.ts.

Read fallback narrowly. If config selects embedding or hybrid but no vector store or embedding service is configured, searchMemories() uses the keyword path. This is not a guarantee that every runtime vector/index failure will produce keyword results; those failures are still bounded by timeout and error handling so the main chat can continue, but this turn may have no recall.

Why not inject full scenes automatically

Automatic injection

The automatic path gives a map: scene file path, heat, update time, and summary. It tells the model where full evidence may live without dumping every scene block into the prompt.

Explicit tool search

When an answer needs detail, the agent should call read_file for the relevant scene, or use tdai_memory_search for L1 and tdai_conversation_search for raw L0 evidence.

Scene Navigation is progressive disclosure: index first, full content only on demand. Full scenes can be long and are not relevant to every turn; automatic full injection would crowd out the user request, system rules, and relevant L1 snippets. Absolute paths in navigation let the agent drill down with read_file only when it needs project background, event history, or stage conclusions.

Many providers make prompt caching friendlier for stable prefixes or stable system regions. SQLite、sqlite-vec、FTS、BM25 与混合搜索SQLite, sqlite-vec, FTS, BM25, and hybrid search

This lesson grounds Lesson 23's strategies in the local SQLite store. VectorStore keeps metadata tables, FTS5 virtual tables, and sqlite-vec vec0 vector tables in the same vectors.db: keyword recall uses FTS5/BM25, semantic recall uses cosine distance, and hybrid recall does an RRF-style client-side merge through a shared helper or a local equivalent in the calling path.

🧭 Analogy
Think of a local file cabinet with three indexes. A registry records each document's id, time, and category; keyword cards find documents by words; semantic cards find documents by nearby meaning. The assistant then fuses both candidate lists with an RRF-style merge instead of trusting only one index.

The four local store layers

metadatal1_records / l0_conversations
Ordinary SQLite tables store displayable, filterable, traceable fields: L1 content/type/priority/scene/timestamps, and L0 session, role, message_text, and recorded_at.
keywordl1_fts / l0_fts
FTS5 virtual tables store tokenized index columns plus original text for display; buildFtsQuery() builds the query and bm25() ranks results ascending by rank.
vectorl1_vec / l0_vec
vec0 tables store embeddings and use distance_metric=cosine for nearest-neighbor search; dimensions come from embedding config.
embeddingprovider + model + dimensions
The embedding service records provider/model/dimensions. Provider, model, or dimension changes make old vectors incomparable; store initialization uses embedding metadata to decide when to drop vec tables and reindex.

Write path: one transaction around metadata, FTS, and vectors

write L1
The L1 writer passes a record and optional embedding.
->
metadata insert
l1_records uses ON CONFLICT to update normal fields.
->
FTS tokenize
tokenizeForFts() writes l1_fts; FTS failures only degrade keyword indexing.
->
embedding
When a usable vector exists, it enters vec0; without embedding, metadata + FTS still persist.
->
vector upsert
vec0 does not support ON CONFLICT, so upsert is delete then insert.

sqlite.ts uses manual BEGIN/COMMIT and ROLLBACK on failure, keeping metadata, FTS, and vector rows synchronized for a record as much as possible. The database runs in WAL mode for safer concurrent reads and writes. Remember that FTS and vec0 are derived indexes; the metadata tables are the traceable source records.

Three recall columns

BM25 keyword recall

buildFtsQuery(query) turns user input into an executable FTS5 query; bm25(l1_fts) or bm25(l0_fts) ranks matches. It is strong for exact terms, function names, file names, and tokenized Chinese keywords.

Vector semantic recall

The query is encoded by the same embedding provider, then nearest neighbors are searched in l1_vec/l0_vec by cosine distance. This helps when users ask with different wording.

RRF-style hybrid

Local SQLite has no server-side native hybrid rank. Tool code collects FTS and vector candidates separately, then uses a shared RRF helper or a local equivalent in the calling path to fuse rankings into a more robust order.

Core pseudocode

writeMemory(record):
    begin_transaction()
    upsert(l1_records, record.metadata)
    upsert(l1_fts, tokenize_for_fts(record.content))
    if embedding_available:
        delete(l1_vec where record_id = record.id)
        insert(l1_vec, embed(record.content))
    commit()

hybrid(query):
    ftsRanked = searchL1Fts(buildFtsQuery(query))
    vecRanked = searchL1Vector(embed(query))
    return rrf_style_merge_or_local_equivalent([ftsRanked, vecRanked])

Provider metadata and reindexing

Vectors are not plain text indexes; they are bound to provider, model, and dimensions. The same text embedded by different models may live in incompatible vector spaces, and different dimensions make the vec0 schema incompatible. So when embedding_meta detects provider/model/dimensions changes, the store preserves l1_records and l0_conversations, rebuilds l1_vec/l0_vec, and reindexes with the current embedding service. A local provider may need warmup; a remote provider is HTTP-based, but both must emit vectors matching configured dimensions.

腾讯云 VectorDB 后端与 Embedding 服务Tencent Cloud VectorDB backend and embedding services

Lesson 25 explained the local SQLite store; the final M3 lesson switches to the remote backend. TcvdbMemoryStore writes L1 and L0 into Tencent Cloud VectorDB hybrid collections prefixed by the database name: dense embeddings use the tcvdb.embeddingModel 1024-dimensional index, sparse vectors are generated client-side by the BM25 encoder, and native hybridSearch with RRF rerank combines recall. Profiles use a separate sync/query collection rather than hybrid recall; if remote access fails, JSONL and local evidence files still keep memory traceable and recoverable.

🧭 Analogy
SQLite is a portable file cabinet with all indexes maintained locally; TCVDB is a cloud archive where registration, vector retrieval, and hybrid reranking can happen server-side. The assistant still carries raw receipts: even if the cloud archive is temporarily unavailable, local JSONL and evidence files do not disappear.

Local SQLite versus cloud TCVDB

SQLite local backend

Metadata, FTS5, sqlite-vec, and JSONL are local. Hybrid search usually collects keyword/vector candidates separately and does client-side RRF-style merging. It is cloud-free and offline-friendly; its indexing and concurrency follow the local machine.

Tencent Cloud VectorDB backend

L1/L0 collections, dense embedding fields, sparse vectors, filter indexes, and native hybridSearch move into VectorDB; the profiles collection only does metadata/profile sync queries. It provides centralized remote retrieval, server-side RRF rerank, and migration/export paths; it requires valid config, network, and credentials.

Degradation, migration, and export

The remote backend is not a hard dependency of the main chat, but storeBackend: "tcvdb" does not keep automatic SQLite/local fallback search. factory.ts selects the TCVDB bundle and returns NoopEmbeddingService; tcvdb-client.ts keeps HTTP errors, timeouts, and response parsing at the client boundary. When TcvdbMemoryStore enters degraded state, remote search may return an empty array and remote upsert/delete may no-op instead of switching to local SQLite queries.

Degradation does not mean evidence disappears. L0 JSONL, L1 evidence files, scene blocks, and persona/profile file paths produced by capture/extraction should remain available, so the next turn can keep auditable local evidence or wait for remote recovery. Existing SQLite data can be migrated with scripts/migrate-sqlite-to-tcvdb/sqlite-to-tcvdb.ts; remote collections can be exported with scripts/export-tencent-vdb/export-tencent-vdb.ts, keeping rollout and rollback auditable.

为什么长任务日志需要符号压缩Why long task logs need symbolic compression

Tool logs in long tasks grow quickly: the same file is read again and again, command output fills screens, and early results become stale while still occupying live context. Context Offload does not replace L0-L3 long-term memory. It adds a current-task symbolic layer: raw output goes to refs files, compact rows go to offload JSONL, task state is rendered as Mermaid MMD, and the next turn injects only the symbols needed to continue.

🧭 Analogy
L0-L3 is a cross-project archive that helps future sessions remember facts, scenes, and profiles; Context Offload is the whiteboard on your desk. The whiteboard does not keep every detail in view. It keeps “where is the evidence, where are we now, which node should we inspect next” visible.

Why long task logs crowd the window

Repeated tool calls

Coding sessions repeatedly run rg, read files, and run tests. Each result may enter message history even when the next step only needs the symbol “this file was checked”.

Large command output

Builds, tests, and logs can produce hundreds or thousands of lines. The raw output is evidence, but the next reasoning step usually needs a failure summary, reference path, and current state.

Stale results remain

Early search hits, old stack traces, and fixed failures keep consuming tokens. They should remain traceable, but they should not permanently crowd the real-time prompt window.

Symbolic compression flow

long task logs
Tool calls, parameters, stdout/stderr, stack traces, and duration.
->

Full evidence is written under refs/ with a recoverable path.
->
Offload-L1 JSONL
OffloadEntry stores summary, result_ref, tool_call_id, and node_id.
->
Mermaid MMD
Offload-L2 organizes JSONL rows into task nodes and status canvas.
->
Offload-L3 injected context
Only active task symbols are injected, not the whole tool log.
⚠️ Naming boundary
Context Offload's internal Offload-L1/L2/L3 stages are short-term pipeline stages, separate from long-term L0-L3 memory: Offload-L1 roughly means summarization, Offload-L2 means the Mermaid task canvas, and Offload-L3 means context injected into the next turn. Long-term L2/L3 still mean scene blocks and persona/profile.

Long-term L0-L3 versus short-term Context Offload

Long-term L0-L3 memory

Future-session oriented: L0 stores raw conversation evidence, L1 extracts atomic memories, L2 organizes scene blocks, and L3 generates persona/profile context. The goal is cross-session reuse, search, and long-term evolution.

Short-term Context Offload

Current-task oriented: move tool results out of live context into local evidence files and a task symbol layer. The goal is long coding-session continuity, drill-down recovery, and controlled token budget.

Shared principle, different boundary

Both care about evidence paths and compressed summaries. The lifecycle differs: Offload solves current-task navigation, not cross-session long-term factual memory.
