为什么需要 Agent MemoryWhy Agent Memory

TencentDB Agent Memory 解决的是一个长程 Agent 的老问题：Agent 做得越久，对话、工具日志、项目规则和用户偏好越多， 上下文窗口却始终有限。这个项目不是把所有历史都粗暴塞回 prompt，而是把信息分层保存、分层召回，让 Agent 既能记住长期经验， 又能在长任务里把临时日志压缩成可追溯的结构。

🔌 生活类比
把 Agent 的上下文想成一张办公桌。直接把所有文件堆上桌，很快就看不见重点；TencentDB Agent Memory 像一个分层档案柜： 桌上只放当前任务地图，抽屉里放摘要，档案室里保留原文，需要证据时再按编号下钻。

两个记忆问题

长期记忆

跨会话记住偏好、事实、场景和用户画像，下一次对话前按需召回。

短期压缩

单个长任务中工具日志太大时，把原文卸载到文件，只把 Mermaid 任务画布留在上下文。

“上层保结构、下层保证据”是读这个项目时最重要的抓手。L3 Persona 和 L2 Scene 把大量历史压成可读、可注入的结构， 让模型先看到稳定结论；L1 Atom 和 L0 Conversation 则保留更细的事实与原始对话，方便在结论可疑时逐层下钻核对。 所以记忆不是只追求更短摘要，而是在“回答时够精炼”和“追责时有证据”之间建立一条可返回的路径。

项目的两条主线

L0Conversation

一次对话的数据流The data flow of one conversation

这一课把镜头拉近：一轮真实对话从宿主框架进入插件后，如何被捕获、抽取、聚合，并在下一轮对话前重新注入。 先看数据流，后面课程再逐段拆源码。

🔌 生活类比
像快递分拣：入口先登记包裹，流水线按规则抽取信息，仓库按主题归档，下一次需要时按收件地址和关键词把相关包裹找出来。

完整路径

1. before prompt读取 Persona、Scene Navigation，并按用户输入搜索 L1。
2. model turn宿主框架运行主 Agent。
3. turn committed对话结束后，插件捕获新增 user/assistant 消息。
4. L0 write写入 conversations JSONL，并更新 checkpoint。
5. pipeline调度 L1 抽取、L2 场景、L3 Persona。

按时间线看，只有 recall/search 属于模型回答前的准备阶段：它把已经沉淀的 Persona、Scene 导航和相关 L1 记忆放到本轮上下文里， 帮助模型带着“该记得的东西”开始回答。模型回答之后，capture 才能拿到完整的新消息并写入 L0；随后 pipeline 再异步或按触发条件抽取 L1、 合并 L2、更新 L3。这样设计避免了把尚未完成的回答提前写成事实，也让每一轮都遵循“先召回旧证据，再沉淀新证据”的节奏。 如果排查召回问题，也应按这个顺序定位：先看回答前是否查到了合适记忆，再看回答后是否成功捕获并进入管线。 前后阶段分清后，日志、文件和代码入口才不会混在一起。

两条记忆主线Two memory spines

TencentDB Agent Memory 不是一条“万能记忆流水线”，而是两条协作的记忆主线：长期 L0-L3 负责把可复用经验沉淀到未来会话， 短期 Offload 负责把当前长任务的工具结果压缩成可浏览、可下钻的上下文地图。两条线都保留证据，但优化目标完全不同。

🔌 生活类比
长期记忆像团队知识库：把反复有用的规则、偏好、场景和画像整理好，下一次项目还能用。短期 Offload 像这次会议的临时白板： 工具输出、报错、文件片段都先贴到白板背后的附件里，桌面只保留结构化索引，讨论结束后不必把每个临时细节都变成长期偏好。

两条主线的分工

长期记忆 L0-L3

从原始对话开始，逐层抽取 Atom、聚合 Scene、生成 Persona，再在下一轮或下一次会话前召回。它回答“哪些经验值得以后继续使用”。

短期 Offload

从工具结果开始，写入 refs markdown 和 offload JSONL，先由 Offload-L1 产出紧凑摘要行，按需用 Offload-L1.5 判断并精炼近期任务上下文，再由 Offload-L2 生成 Mermaid MMD 画布并分配 node_id，最后通过 Offload-L3 注入压缩符号上下文。它回答“当前任务如何少占 prompt”。

⚠️ 名称边界
长期 L2/L3 仍然指 Scene 场景块和 Persona 用户画像；Offload-L2/L3 是短期上下文卸载的内部阶段，分别指 Mermaid MMD 任务画布和压缩符号注入。

Offload 的注册和上下文引擎看 src/offload/index.ts 中的 registerOffload 与 OffloadContextEngine。证据落盘由 src/offload/storage.ts 的 writeRefMd、 appendOffloadEntries、createStorageContext 负责；是否该提升到 Offload-L2 Mermaid 结构，可看 src/offload/pipelines/l2-mermaid.ts 的 checkL2Trigger。这条线不是为了长期学习用户，而是为了让当前任务在工具输出爆炸时仍能保住结构。

共同原则：上层保结构，下层保证据

上层结构

L3 Persona、L2 Scene、Offload-L2 MMD 和 Offload-L3 注入都应该短、稳定、可导航，让模型先看到“地图”。

下层证据

L0 对话、refs markdown、offload JSONL 保存原文、来源和 node_id，让 Agent 只在需要确认细节时下钻。

为什么不能合并成一条线

如果把 Offload 都塞进长期记忆，临时工具日志会污染未来会话；如果把长期偏好只放在 Offload，任务结束后又无法稳定复用。 长期 L0-L3 优化的是跨会话复用：更关心事实是否长期有效、是否能帮助下一次决策。短期 Offload 优化的是当前上下文压力： 更关心如何把大量工具输出变成一张轻量任务图，并保留足够证据让模型按 node_id 回看原文。两条主线协作，而不是互相替代。

OpenClaw 零配置闭环OpenClaw zero-config loop

本课把前面讲过的 L0-L3、Recall 和搜索工具接到一个最小可运行闭环里：安装 OpenClaw 插件、打开最小配置、重启 Gateway， 再用几轮对话先验证 L0 捕获和 L1 搜索/召回。这里的目标不是调优，而是先证明“能跑、能存、能找、能在新回合被想起”；L2/L3 可能稍后出现。

🔌 生活类比
零配置像给电脑接上一块默认本地硬盘：你不用先规划云数据库、索引参数或复杂密钥，就能保存和读取文件。 但如果要做更远距离、更大规模或更强向量召回，就要再接入真实 embedding、TCVDB 等高级配置。

一条最小闭环

1

安装或更新插件

使用 OpenClaw 插件命令安装 @tencentdb-agent-memory/memory-tencentdb；升级时显式运行同名 update 命令。

openclaw plugins install @tencentdb-agent-memory/memory-tencentdb
openclaw plugins update @tencentdb-agent-memory/memory-tencentdb

2

启用最小配置

在 ~/.openclaw/openclaw.json 中只打开 enabled。不要在教程示例里写真实 API Key。

3

重启 Gateway

插件配置、hooks 和 tools 注册发生在 Gateway 生命周期里，改完配置后要重启才能稳定生效。

openclaw gateway restart

4

对话 2-3 轮

告诉 Agent 一个可复用偏好或项目约束，再继续追问，让自动捕获先沉淀到 L0 原文与 L1 结构化记忆。

5

验证文件与召回/搜索

先确认 ~/.openclaw/state/memory-tdai/ 下出现 conversations/、records/、scene_blocks/ 或 vectors.db 等产物，再验证 L0 捕获、L1 搜索/召回；L2 场景块和 L3 persona 可能需要管线延迟或触发条件才会出现。

最小配置就是这么少

{
  "memory-tencentdb": {
    "enabled": true
  }
}

“零配置”的意思是：只要启用插件，默认会使用本地 SQLite 与 sqlite-vec 路径保存数据，适合课程、个人试用和离线 smoke test。 它不是“所有能力都自动最佳”：如果没有配置可用的 embedding，向量行为可能被默认关闭或退化，搜索更多依赖关键词、结构化层级和已有索引。 所以最小配置先证明闭环，高级配置再处理召回质量、并发、远程存储、模型和凭据管理。

日常最小配置 vs 高级调优

日常最小配置

只写 enabled: true，用默认本地 SQLite + sqlite-vec 路径。适合先验证 OpenClaw 能注册插件、能写入 L0 原文、能抽取并搜索/召回 L1；L2 场景块和 L3 persona 可能稍后在管线延迟或触发条件满足后出现。

高级调优

再按需要配置 embedding、远程数据库、TCVDB、保留策略、调试日志和性能参数。真实 API Key 只放在本机安全配置或环境变量中，不写进公开教程、截图、提交或示例代码。

Hermes 与 Gateway 路径Hermes and Gateway path

Hermes 路径看起来和 OpenClaw 不同：一个是 Python Agent 的 memory provider，一个是 OpenClaw 插件和 Gateway 生命周期。 但它们不是两套记忆引擎。Hermes 只是另一条宿主入口：provider 把 Hermes 的 recall、capture、search 调用转成 HTTP，交给本地 Gateway，Gateway 再通过 StandaloneHostAdapter 调到同一个 TdaiCore。

🚪 生活类比
OpenClaw 和 Hermes 像从两扇不同的门进入同一座图书馆。门口的登记员不同：OpenClaw 用插件 hooks/tools，Hermes 用 memory_tencentdb provider。 但进入馆内后，借书、归档、检索和整理书架都由同一个核心服务处理，所以学习重点是“入口不同，核心收敛”。

在 hermes-plugin/memory/memory_tencentdb/README.md 中，provider 被描述为“薄 HTTP client + process supervisor”：它负责接入 Hermes 生命周期、启动或探测 Gateway、把 prefetch 映射到 POST /recall，把 sync_turn 映射到 POST /capture，并提供搜索工具 schema。 真正的捕获、抽取、存储、召回和管线调度仍在 Node.js Gateway 与核心模块中完成。

两种安装路线：从零 Docker，还是挂到已有 Hermes

Docker 全新部署

README_CN.md 的 Hermes 章节把 Docker 路线定位为“从零启动一个带记忆能力的 Hermes”。镜像把 hermes-agent 和 memory_tencentdb provider 聚合在一起，Gateway 暴露在 8420 端口。适合想先用一条容器路线验证完整能力的人。

已有 Hermes 安装 provider

如果机器上已经有 Hermes，就把 provider 链接或复制到 Hermes 的 memory provider 目录，并在 ~/.hermes/config.yaml 中声明 memory.provider: memory_tencentdb。目录名必须正好是 memory_tencentdb，因为 Hermes 用目录名作为 provider key。

无论从 Docker 进入，还是把 provider 挂到已有 Hermes，关键检查点都一样：Hermes 能发现 provider，provider 能连到本地 Gateway，Gateway 的 HTTP 路由能到达 TdaiCore。 这也是为什么教程不把 Hermes 讲成“另一套内存系统”：它改变的是宿主集成和进程边界，不改变 L0-L3、Recall/Search 与管线核心。

Gateway 对外暴露什么

src/gateway/server.ts 标明 Gateway 是 Hermes sidecar 的 HTTP server：GET /health 用于健康检查，POST /recall 做模型回答前的记忆召回，POST /capture 捕获一轮对话，POST /search/memories 搜索 L1 结构化记忆，POST /search/conversations 搜索 L0 原始对话。 src/gateway/config.ts 还提供 server.apiKey / TDAI_GATEWAY_API_KEY 和 server.corsOrigins / TDAI_CORS_ORIGINS 这类可选开关；默认重点是本地 sidecar，跨网络开放前才需要认真收紧鉴权与 CORS。

运行后会出现哪些文件Runtime files and data products

运行后的文件不是隐藏实现细节，而是一张调试地图：当召回、画像、场景或长任务摘要看起来不对时，先看高层产物理解系统“相信了什么”，再一路下钻到低层证据确认它为什么这么相信。

🗺️ 生活类比
这些目录像侦探案卷：画像和场景是案情摘要，L1 原子记忆是证词卡片，L0 对话是原始录音；Offload 的 MMD 是任务白板，JSONL 和 refs 是白板背后的材料袋。

L0-L3 分层会落到哪些文件

L0Conversation
conversations/ 保存原始对话 JSONL，由 src/core/conversation/l0-recorder.ts 写入，负责保留最底层证据。
L1Atom
records/ 保存结构化记忆 JSONL，由 src/core/record/l1-writer.ts 写入，便于搜索和召回。
L2Scene
scene_blocks/ 保存 Markdown 场景块，由 src/core/scene/scene-extractor.ts 聚合多条 L1 证据。
L3Persona
persona.md 保存长期画像，由 src/core/persona/persona-generator.ts 从场景中生成稳定偏好和用户特征。

本地长期记忆目录通常还会看到 vectors.db：它是本地 SQLite 向量/全文检索存储，入口可从 src/core/store/factory.ts 看存储后端如何创建。 .metadata/ 放 manifest、索引状态和检查点线索，对应 src/utils/manifest.ts 与 src/utils/checkpoint.ts；.backup/ 用于备份，方便升级或调试前回滚。 所以调试时不要只盯一个数据库文件：目录、JSONL、Markdown、metadata 和 backup 一起构成“可解释的记忆状态”。

长期记忆目录 vs Offload 目录

长期记忆：跨会话复用

memory-tdai/ 位于解析后的 OpenClaw state dir 下，常见完整路径是 ~/.openclaw/state/memory-tdai/；standalone 或 fallback 路径可能不同。里面的 conversations/、records/、scene_blocks/、persona.md、vectors.db、.metadata/、.backup/ 关注“未来会话还能不能记得、搜到、解释清楚”。它的证据链是 L3 -> L2 -> L1 -> L0。

Context Offload：当前任务压缩

~/.openclaw/context-offload/<agent>/ 下会有 refs/、mmds/、offload-<session>.jsonl、state.json、sessions-registry.json。它关注“当前长任务如何把大块材料移出上下文，但仍能找回证据”。实现入口看 src/offload/storage.ts。

最小文件形状

memory-tdai/
  conversations/     # L0 raw messages
  records/           # L1 atoms
  scene_blocks/      # L2 scenes
  persona.md         # L3 persona
  vectors.db         # local search index

context-offload/
  <agent>/refs/
  <agent>/mmds/
  <agent>/offload-<session>.jsonl

Offload 的 refs/ 保存被卸载的大块原文或引用材料，mmds/ 保存 Mermaid/MMD 任务图，offload-<session>.jsonl 记录会话内的卸载事件。 state.json 维护当前状态，sessions-registry.json 则帮助按会话找到对应产物。MMD 适合先看全局任务结构，但如果节点描述和真实材料冲突，最终仍要回到 JSONL 与 refs。

下钻调试：先看结构，再找证据

1

Persona -> Scene

如果用户说“画像不对”，先打开 persona.md 看具体 claim，再找它来自哪些 scene_blocks/。

场景块只保结构和叙述，不是最终证据。继续到 records/ 找 L1 Atom，核对事实、偏好、约束的抽取是否合理。

配置层级与安全默认值Configuration levels and safe defaults

配置不是越多越专业。TencentDB Agent Memory 的安全心智模型是：先让零配置闭环能跑，再只调日常旋钮，最后才进入长会话、远程后端、LLM 与运维边界。 本课把 src/config.ts、openclaw.plugin.json、README_CN.md 的“可调参数”说明和 src/core/store/factory.ts 串起来，形成一张渐进式配置地图。

🎚️ 生活类比
配置像调音台：日常只动音量和静音，长会话才调均衡器和压缩器，真正换声卡或接远程设备时才碰驱动、接口和凭据。默认值应该让新手先听到声音，而不是先面对一整排危险旋钮。

三层配置：从日常旋钮到运维边界

1Daily knobs
日常只调 capture、recall、少量 pipeline：是否捕获、召回条数、召回策略、超时和最小触发条件。目标是“别打扰用户，还能想起来”。
2Long-session tuning
长会话再看 extraction、pipeline、offload、bm25：抽取频率、批处理、关键词检索、上下文卸载和报告产物。目标是减少上下文压力，同时保留可追溯证据。
3Backend / LLM / ops
最后才配置 embedding、tcvdb、llm、report 和网关鉴权。这里涉及远程服务、模型维度、API Key、后端选择和运维可观测性。

这三层不是权限等级，而是学习顺序。parseConfig 会把用户配置和默认值合并成运行时配置；因此文档应鼓励“少量、可解释、可回滚”的改动。 如果一开始就同时改 embedding、TCVDB、清理策略和 LLM，很难判断问题来自网络、凭据、维度、后端还是业务触发条件。

关键配置组

  • capture：控制对话是否进入 L0/L1 流程，以及 l0l1RetentionDays 这类保留策略；值为 0 表示关闭清理，不会自动删 L0/L1。
  • extraction：影响从原始对话抽取结构化记忆的方式，适合在确认捕获稳定后再调。
  • pipeline：PipelineTriggerConfig 这类触发配置决定何时整理 L1、场景、画像或批处理任务。
  • recall：RecallConfig 管理策略、最大条数和 timeoutMs；超时是安全默认值，防止召回卡住用户请求。
  • embedding：EmbeddingConfig 决定本地或远程向量能力；远程 provider 必须给出 apiKey、baseUrl、model、dimensions。
  • bm25：关键词检索补足纯向量召回的盲区，适合术语、文件名、错误码和专有名词。
  • tcvdb：远程 Tencent Cloud VectorDB 后端配置；只有同时设置顶层 storeBackend: "tcvdb"，后端才会切到 TCVDB。
  • offload：OffloadConfig 管理长任务材料外置、refs、MMD 和会话状态，服务当前任务压缩。
  • llm：控制抽取、总结或网关相关模型调用；真实密钥必须留在安全本地配置或环境变量中。
  • report：面向调试和可读产物，帮助人查看发生了什么，而不是替代底层证据链。

本地默认与 TCVDB 后端

SQLite default

零配置预期可运行：默认本地 SQLite / sqlite-vec 路径适合课程、离线试用和个人 smoke test。它降低启动门槛，也让新手先验证 capture、pipeline、recall 的闭环。清理策略默认保守：capture.l0l1RetentionDays = 0 表示禁用清理；任何激进清理都应显式 opt-in。

TCVDB backend

当数据规模、共享部署或远程向量检索成为真实需求，再配置 tcvdb。不要只填 tcvdb 字段：还必须在 memory-tencentdb 插件块的顶层设置 storeBackend: "tcvdb"，store bundle 才会走远程后端。

TCVDB 路径需要两个部分同时出现在 memory-tencentdb 插件块中：顶层后端选择和 tcvdb 连接占位符。示例只使用假 URL 与环境变量名：

{
  "memory-tencentdb": {
    "storeBackend": "tcvdb",
    "tcvdb": {
      "url": "https://example.invalid",
      "apiKey": "${TCVDB_API_KEY}",
      "database": "my-memory-db"
    }
  }
}

TdaiCore：宿主无关核心门面TdaiCore: host-neutral facade

TdaiCore 是 OpenClaw 插件路径与 Gateway/Hermes 路径共用的宿主无关核心门面。 宿主差异由 adapter 吸收；core 对外稳定暴露自动召回、对话提交捕获、记忆搜索、对话搜索与 pipeline scheduler 生命周期。

🧩 生活类比
如果 OpenClaw 与 Gateway/Hermes 是不同入口大厅，adapter 就是前台翻译，TdaiCore 是同一个资料室柜台。 不管访客从哪个大厅来，最终都向同一个柜台办理“查资料、登记新资料、维护后台整理任务”。

三层边界：宿主、Adapter、Core

HostOpenClaw / Gateway
OpenClaw hooks/tools 或 Gateway/Hermes HTTP 调用，负责宿主生命周期与请求形态。
AdapterHost adapters
把日志、路径、LLM runner、运行上下文翻译成 core 可用的统一接口。
CoreTdaiCore
统一处理 recall、capture、memory search、conversation search 与 scheduler。

这个边界让核心能力可以被复用：OpenClaw 的 before/after hook 不需要复制 Gateway 的记忆逻辑；Gateway/Hermes 也不需要理解 OpenClaw 的插件注册细节。两条路径都把请求变成 TdaiCore 能理解的方法调用。

生命周期先于能力调用

1

initialize()

创建数据目录，调用 initStores 准备向量/embedding 存储，并用 createPipelineManager 构造 scheduler。

2

在模型回答前转交给 performAutoRecall，返回要注入上下文的召回片段。

3

handleTurnCommitted

对话提交后先确保 scheduler 已启动，再调用 performAutoCapture 写入 L0 并触发后续 pipeline。

4

scheduler / destroy()

后台 pipeline 持续整理 L1-L3；关闭时 destroy() 先排空后台任务，再关闭 store。

两条宿主路径复用同一个门面

OpenClaw path

index.ts 注册 hooks 和 tools。before hook 调 handleBeforeRecall；after/agent_end 提交完整 turn 给 handleTurnCommitted；工具调用转到 search 方法。

Gateway / Hermes path

Gateway HTTP handler 接收 Hermes provider 的请求，同样调用 handleBeforeRecall、handleTurnCommitted 和搜索方法，因此不复制核心算法。

核心伪代码

    initDataDirectories()
    storeReady = initStores()
    scheduler = createPipelineManager()

    return performAutoRecall(user_text)

turn_committed(messages):
    await ensureSchedulerStarted()
    return performAutoCapture(messages)

为什么 schedulerStartPromise 是并发闸门

Gateway 场景中，多个 HTTP 请求可能同时进入 handleTurnCommitted。 如果只用布尔值标记“scheduler 已启动”，第一个请求可能在异步读取 checkpoint 尚未完成时就把标记改成 true；后续请求会越过启动流程，提前更新 scheduler 状态。 当真正的 scheduler.start(restoredStates) 稍后执行时，恢复出的旧状态可能覆盖这些并发更新。

schedulerStartPromise 把“正在启动”本身保存为同一个 promise。所有并发请求都 await 同一段启动过程；启动成功后，已 resolve 的 promise 又成为廉价哨兵，后续调用不会重复 start。

为什么要登记后台任务

performAutoCapture 可能为 L0 embedding 发起延迟后台写入。它们是 fire-and-forget，但并不代表可以在关闭时忽略。 TdaiCore 用 background task registry 记录这些 in-flight promise；destroy() 会在关闭 vector store 与 embedding service 前等待它们完成或超时，避免迟到写入打到已关闭数据库连接。

运行时初始化与配置解析Runtime init and config parsing

本课把前几课的“入口壳、adapter、core”串到启动顺序上：OpenClaw 给插件一份原始 api.pluginConfig， index.ts 每次注册只解析一次，先准备配置、时间模块和目录；TdaiCore.initialize() 随后启动异步 store 初始化，但不会等 store 全部完成才创建 pipeline manager。

🧩 生活类比
开店前先看当天排班表：店长只读一次排班表，把默认岗位补齐，先开门并准备货架；清点库房可以在后台继续，收银系统也能先接线。 运行时初始化也是这样：配置和目录先稳定下来，store 初始化异步推进；需要 store 的路径再等待 storeReady，或在 store 不可用时降级。

启动主线：原始配置到 hooks

1

raw plugin config

index.ts 从 api.pluginConfig 取得原始对象，并记录收到的 key 数量。

2

parseConfig

src/config.ts 把用户配置与默认值合并成 MemoryTdaiConfig，本次 register 后续都复用它。

3

init dirs

resolveOpenClawStateDir 选出 OpenClaw state 目录；index.ts 先创建 conversations、records、scene_blocks、.metadata 和 .backup。

4

core starts

core.initialize() 再确认目录，立即把 storeReady = initStores() 放到后台，并同步创建 pipeline manager。

5

hooks / tools

index.ts 在 core 初始化启动后注册 hooks/tools；store 依赖路径在调用时 await storeReady，pipeline runners 在 store 完成或失败后接线。

注意“每次注册解析一次”并不等于全进程只解析一次。OpenClaw 可能因为插件扫描、Gateway 启动、channel bootstrap 或配置 reload 多次调用 register(api)；每次调用都收到完整配置，所以入口直接解析当前 raw object，避免复用过期配置。

零配置与高级配置的差别

零配置

parseConfig(undefined) 或 {} 仍会启动：capture、extraction、pipeline、recall 默认启用；SQLite 是默认 store；embedding provider 为 none，因此向量能力禁用但主对话不被阻塞；offload 默认关闭。

高级配置

用户可以按组覆盖 capture、extraction、pipeline、recall、persona、embedding、offload、store、report、llm、bm25、tcvdb 等行为。远程 embedding 字段不完整时，配置保留错误消息并降级为无 embedding。

配置组分别喂给谁

Entrytimezone / report / offload
入口先用 timezone 初始化时间模块，用 report 控制上报行为，并仅在 offload.enabled 为 true 时注册 Context Offload。
Corecapture / extraction / pipeline / recall / persona
这些组决定是否捕获 L0、如何抽取 L1、scheduler 何时跑、召回数量/超时/策略，以及 L2/L3 画像生成节奏。
StorestoreBackend / embedding / tcvdb / bm25
storeBackend 决定 SQLite 还是 TCVDB；SQLite 再按 embedding 配置决定是否创建远程 embedding service；TCVDB 使用服务端 embedding 与可选 BM25。

目录、manifest 与 checkpoint 为什么先于 store 完成态

initDataDirectories(dataDir) 先把运行时目录建好，后续 L0 JSONL、L1 records、scene blocks、metadata 和备份才有稳定落点。 进入 TdaiCore.initialize() 后，initStores 以 promise 形式启动：它会维护 manifest、创建 store bundle，并在失败时记录降级。 同时 createPipelineManager 可以同步创建，因为 runner 接线被挂到 storeReady.then(...).catch(...) 后面。

因此“目录先准备”是真的；“store 一定先于 pipeline manager、hooks 和 tools 完成”不准确。自动 recall、capture、session end 等 store 依赖入口会 await storeReady；搜索工具则使用当前可用的 store 句柄，缺失时走无向量/空结果等降级路径。

伪代码

TdaiCore.initialize():
    initDataDirectories(dataDir)
    storeReady = initStores()      # async; do not await here
    scheduler = createPipelineManager()  # sync; no store needed
    storeReady.then(wirePipelineRunners).catch(wireDegradedRunners)

    await storeReady.catch(ignore)
    run recall or capture with available stores

这段伪代码省略了日志、hook policy 和 adapter 细节，但保留了关键顺序：先把 raw config 固化为 cfg，再准备运行目录； 然后 TdaiCore 启动异步 storeReady，pipeline manager 可先创建，hooks/tools 在 core 初始化启动后注册。offload 是旁路能力：只有 cfg.offload.enabled 为真，入口才调用 registerOffload(api, cfg.offload)。

Auto Capture：对话提交后的捕获Auto Capture after a turn commits

当一轮对话已经提交（agent_end / turn committed）后， performAutoCapture 负责把本轮新增的用户与助手消息可靠落到 L0。 它先做最短、最确定的捕获路径：写本地 L0，再尽量建立搜索索引，最后通知 pipeline scheduler；是否触发 L1 抽取由 scheduler 根据阈值、idle timer 和状态决定。

🧩 生活类比
对话提交后像收银小票已经打印出来：Auto Capture 先把小票存进本地账本，避免丢单；再把小票编号放进索引柜，方便以后搜索；最后提醒后台整理员“有新账了”。 至于后台整理员现在就归档，还是等凑够一批再处理，是调度规则决定的，不是收银台临时决定。

提交后的捕获主线

1

agent_end / turn committed

宿主把完整 messages 交给 performAutoCapture，此时模型回答已产生，捕获不再影响本轮生成内容。

2

checkpoint lock

CheckpointManager.captureAtomically() 持锁读取 cursor、调用 recordConversation、推进 cursor，避免并发 agent_end 读到同一个旧游标。

3

record L0

recordConversation 从本轮消息中抽取 user / assistant，按位置切片与时间游标过滤，清洗后写入 conversations JSONL。

4

index

如果 vector / FTS store 可用，把 L0Record 元数据写入搜索索引；不可用时不阻断本地 L0 记录。

5

notify scheduler

调用 scheduler.notifyConversation(sessionKey, [])。Capture 只通知“有新对话轮次”，L1 何时跑由 scheduler 决定。

一次 messages 如何变成 L0Record

messages
agent_end 提供的会话历史
->
filteredMessages
只保留新增 user / assistant，清理召回包裹与结构噪声
->

关键是“新增”：recordConversation 优先使用 before-prompt 阶段缓存的 originalUserMessageCount 做位置切片， 只看本轮 prompt build 之后追加的消息；如果这个缓存不可用，captureAtomically 会在原子区内读取 checkpoint 自己维护的 afterTimestamp 游标并传给回调过滤。 调用方不在临界区外提供 cursor；checkpoint 在同一个锁内负责 cursor 的读取与推进。因此 Auto Capture 不是把整段历史反复写入 L0，而是把提交后新出现的消息增量写入。

同步路径 vs 延迟 embedding 路径

同步 embedding / VDB 路径

当 store 不支持延迟 embedding 时，capture 会在 upsert L0 前尽量同步生成 embedding，再把 L0Record 和向量一起写入后端。这适合需要一次性提交向量的远程或 VDB 后端。

延迟 embedding / SQLite 路径

当 store 支持 supportsDeferredEmbedding，capture 先写 metadata + FTS，马上返回主流程；embedding batch 和 updateL0Embedding 在后台补齐，避免 agent_end 被 多秒级向量调用阻塞。

伪代码

performAutoCapture(messages):
    checkpoint.captureAtomically(sessionKey, pluginStartTimestamp, (afterTimestamp) => {
        filtered = recordConversation(messages, afterTimestamp)
        return { maxTimestamp: maxTimestamp(filtered), messageCount: filtered.length }
    })
    ...
    scheduler.notifyConversation(sessionKey, [])

为什么 capture 不直接抽取 L1

performAutoCapture 的职责是可靠记录和通知。它不知道当前会话是否达到 everyNConversations， 也不应该复制 idle timeout、warm-up、重试和 shutdown flush 等策略。MemoryPipelineManager.notifyConversation 更新会话计数、持久化状态，并在达到阈值时 enqueue L1；未达到阈值时重置 idle timer。这样捕获路径更短，调度策略集中在 pipeline manager。

L0 JSONL：原始对话证据层L0 JSONL: raw conversation evidence

L0 不是摘要层，而是“原始对话证据层”：它把可追溯的 user / assistant 消息按 JSONL 追加到本地文件中。 recordConversation 只负责从原始 messages 中找出本次新增内容，清洗召回包裹、元数据和结构噪声，再一行一条写入当天文件。

🧩 生活类比
L0 像法庭证物袋，不像会议纪要。证物袋要保存“当时看到的原文”，方便以后核查；纪要可以提炼观点，但不能替代证物。 所以 L0 先写原文账本，L1/L2/L3 才在后面从账本中抽取结构化记忆。

L0 的边界：证据，不是结论

L0 raw evidence

保存单条消息、角色、时间、会话标识与清洗后的正文。它强调可追加、可 grep、可流式读取、可回放排查。

L1 extracted memory

从多条 L0 中抽取事实、偏好、任务片段或经验。它是解释后的结果，必须能回到 L0 找证据。

清洗和“还原原始 prompt”

清洗后再写入

src/utils/sanitize.ts 的 sanitizeText 会移除注入的 memory/context 包裹、召回标记、元数据、时间戳、媒体/base64 和结构噪声；prompt-injection 与内容质量拒绝推迟到 L1 的 shouldExtractL1。

被 prepend 污染时回退

这一步很关键：L0 要记录用户真正说了什么，而不是“用户消息 + 系统召回上下文”的拼接物。 否则后续 L1 会把 recall 注入的旧记忆误当成本轮新事实，造成证据污染和重复抽取。

Checkpoint：游标、锁与防重复Checkpoints: cursors, locks, and duplicate prevention

Checkpoint 是捕获路径里的“防重复保险”。重复 L0 捕获通常不是因为 JSONL 自己会重复，而是两个 agent_end / Gateway 调用并发进入，或者某次捕获读到了过期 cursor。 本课把 captureAtomically()、position slice、timestamp cursor 和 scheduler 启动门放在同一张图里看。

🧩 生活类比
把 L0 想成仓库入库单。两名库管如果同时看见“上一张单号是 100”，就都可能把 101 号货物登记一遍。 正确做法是先锁住登记簿：读到 100、写入新单、把游标推进到 101，这三步完成后再交给下一位库管。

没有锁时：旧游标会制造重复捕获

1

并发入口

OpenClaw 的 agent_end 和 Gateway capture 可能几乎同时提交同一会话的 messages。

2

读到同一个旧 cursor

如果 cursor 在临界区外读取，两个调用都可能看到 afterTimestamp = T0。

3

重复写 L0

两个 recorder 都认为 T0 之后的消息是新增，于是相同 user / assistant 记录被追加两次。

4

迟到推进

最后才写 checkpoint 已经太晚：重复证据已经进入 L0，后续 L1 也可能重复抽取。

有锁时：checkpoint 文件锁串行化修改

1

captureAtomically(sessionKey)

CheckpointManager 使用 checkpoint 文件级 async lock 串行化修改；sessionKey 用来在锁内选择该会话拥有的 cursor 与状态。

2

read checkpoint

文件锁先让 checkpoint 文件修改串行进入，再在锁内读取/更新 sessionKey 拥有的 cursor 与 checkpoint 状态/计数器；调用方不携带外部旧 cursor。

3

record L0

回调调用 recordConversation，只写 cursor 或位置切片之后真正新增的消息。

4

advance cursor

如果本次写入了消息，就在同一把锁内把 checkpoint 推进到新记录的最大 timestamp。

伪代码

captureAtomically(sessionKey):
    lock(checkpoint_file)
    state = read_checkpoint_file()
    cursor = state.sessions[sessionKey].afterTimestamp
    messages = record_after_cursor(cursor)
    if messages:
        state.sessions[sessionKey].afterTimestamp = max_timestamp(messages)
        write_checkpoint_file(state)
    unlock(checkpoint_file)

position slice 与 timestamp cursor 分工不同

position slice

优先使用 prompt build 前缓存的 message count，例如 originalUserMessageCount。提交后只取这个位置之后的数组切片，能避免同一轮 messages 中 timestamp 漂移、同毫秒或宿主改写时间造成的边界误判。

timestamp cursor

当稳定的原始 timestamp 存在时，checkpoint 中的 afterTimestamp 可作为增量捕获 fallback 和审计线索。若原始 timestamp 缺失或不可靠，这个游标很弱；缺少缓存或进程重启时只能降低重复风险，提供的保证也弱于 position slice 原本能给出的本轮边界保护。

因此二者不是互相替代：position slice 保护“本轮新增范围”的结构边界，通常比 timestamp cursor 更强；timestamp cursor 只在稳定原始时间可用时保护跨调用的持久增量边界。 在缺少位置缓存或进程重启的场景，它提供的是较弱的 fallback，而不是等价保证。

schedulerStartPromise：启动状态不能覆盖并发 capture

Gateway starts
恢复 scheduler 持久化状态
->
schedulerStartPromise
所有调用等待同一个启动 promise
->
capture / notify
启动完成后再通知会话计数
->
no clobber
恢复状态不会覆盖并发捕获刚写入的状态

TdaiCore.schedulerStartPromise 解决的是另一类竞态：Gateway 恢复 scheduler 状态时， 同时到来的 capture 如果直接通知 scheduler，可能先写入新计数，随后又被“刚恢复出来的旧状态”覆盖。 启动 promise 把恢复过程变成一道门，让并发调用共享同一个启动结果，而不是各自抢跑。

L1 抽取：从对话到原子记忆L1 extraction: conversation to memory atoms

L1 是把 L0 原始消息整理成“原子记忆”的抽取层。extractL1Memories 不会把每条聊天都永久升级为记忆；它先通过 shouldExtractL1 应用 L0 结构过滤、纯符号/纯问号过滤和 prompt-injection 检测，再把新消息和少量背景消息交给 LLM，让输出成为可追溯、可去重、可排序的结构化 JSON。

🧩 生活类比
L0 像完整录音，L1 像会议后的索引卡。索引卡不会抄下整段原文，而是按 persona、episodic、instruction 三类留下可复用信息。 但每张卡都要写上来自哪几段录音，之后有人质疑时才能回到原始证据。

L0 消息如何进入 L1 抽取

1

L0 messages

pipeline 读取已捕获的 user / assistant L0 记录，它们仍是消息级证据，而不是结论。

2

quality gate

shouldExtractL1 先复用 L0 的结构过滤，再排除纯符号、纯问号和疑似 prompt-injection 文本；长度过滤在当前实现中是注释掉的。

3

background / new split

尾部新消息是本批抽取对象；前面的背景窗口只提供上下文，避免 LLM 因缺少前因后果而误解。

4

LLM extraction

src/core/prompts/l1-extraction.ts 要求模型按场景分段输出 JSON；parseExtractionResult 去掉 fence 并截取 JSON 数组，随后 sanitizeJsonForParse 修复控制字符再交给 JSON.parse。

5

extracted memories

每个原子记忆带 type、priority、source_message_ids 与 metadata，随后交给 writer 持久化。

L0 原文与 L1 原子记忆

L0 raw message

一条消息记录“谁在什么时候说了什么”。它保留证据，粒度偏细，可能包含寒暄、重复、临时纠错和上下文依赖。

L1 memory atom

一个结构化结论记录“值得未来复用的事实或经验”。它可以来自多条 L0，并用 source_message_ids 指回证据。

L1 字段

字段含义质量作用
type支持的记忆类型：persona、episodic、instruction偏好归入 persona；决定、计划和事件归入 episodic；长期行为规则归入 instruction
content归一化后的最小可复用记忆陈述，不是原文逐字复制避免把整段对话搬进长期记忆，只保存稳定含义
priority重要度或保留价值帮助排序、筛选和后续压缩
source_message_ids贡献该记忆的 L0 消息 ID 列表保留从 L1 回到 L0 的审计链路
metadata主要是 episodic 的活动时间字段（activity_start_time / activity_end_time）；其他类型通常为 {}，如需辅助字段应遵循 schema补充可用的时间范围等上下文；scene 属于上层分段，不是每条记忆的 metadata 字段

为什么同时给 new messages 和 background messages

extractL1Memories 把“新消息”作为本次必须抽取的范围，把“背景消息”作为只读上下文。 这样 LLM 能理解代词、纠错、任务阶段和上一句决定，又不会被整段历史淹没。背景窗口越大，成本和误抽取风险越高；窗口太小，又可能把“好，就按第二个方案”抽成没有意义的孤立句。

qualified = messages.filter(shouldExtractL1)
new_messages = tail(qualified, maxNewMessages)
background = previous_window(qualified)
scenes = call_llm_extraction(new_messages, background)
memories = flatten_scene_memories(scenes)

L1 去重、冲突检测与写入L1 dedup, conflict detection, and write path

L1 抽取会把多条 L0 消息变成结构化记忆，但抽取结果可能重复，也可能与已有记忆冲突。写入前的 batchDedup 是质量控制：先召回相似候选，再判断这条新记忆应该 store、update、merge，还是 skip。 它控制的是可搜索 L1 事实的质量，而不是删除 L0 原文证据。

🧩 生活类比
L0 像会议录音库，L1 像团队知识卡片。录音库不能因为有人做了新卡片就剪掉原声； 去重只是检查“这张卡是不是已经有了、是不是需要修订旧卡、是不是与旧卡冲突”，让知识卡片架保持清晰。

从抽取结果到写入决策

extracted memories
LLM / logic 产出候选 L1 原子记忆
->
candidate recall
向量或 FTS 召回可用时查找相似已有记忆
->
dedup decision
判断 store / update / merge / skip 与冲突说明
->
write / update / merge / skip
按决策把 L1 写入 JSONL 与可搜索 store，或跳过

src/core/record/l1-dedup.ts 中的 batchDedup 会在批量写入前处理已经分配 record_id 的抽取结果，返回 store/update/merge/skip 动作。它优先利用可用召回找候选：同一偏好被重复表达时可 skip； 用户纠正旧偏好时可 update；可合并的信息可 merge；新事实没有相似候选时 store。判断来自 src/core/prompts/l1-dedup.ts 中的 LLM prompt。

为什么去重不是丢弃证据

L0 证据层

保存原始对话 JSONL：谁说了什么、什么时候说、消息 ID 是什么。它服务审计、回放和重新抽取，去重不应改写这层原始证据。

L1 质量层

保存可召回的结构化事实。去重只决定新 L1 是否会让事实库重复、过时或冲突，从而控制未来 recall 的噪声。

写入路径：JSONL 与 store upsert

抽取流程会先用 generateMemoryId 预分配唯一记录 ID：它基于时间与随机字节生成待写入记录的 ID， 不是内容稳定哈希，也不是去重后的语义身份。随后 batchDedup 针对这些 ID 返回决策； writeMemory 使用这条预分配 ID 追加或 upsert L1 记录。 store upsert 路径把同一条记录维护到可搜索后端：本地 sqlite 可提供向量 / FTS 检索，远端 TCVDB 可提供托管向量检索。JSONL 是持久证据，store 是检索索引；两者一起让记忆既可审计又可召回。

SQLite local store

src/core/store/sqlite.ts 适合零配置或本地开发。它在本地数据库维护向量与 FTS 索引，让 L1 记录即使没有远端服务也能被相似搜索或关键词搜索找到。

TCVDB backend

src/core/store/tcvdb.ts 适合接入托管向量数据库。createStoreBundle 根据配置组装 embedding、store 与 fallback，使上层写入逻辑不需要知道具体后端。

召回降级：召回能力下降时仍保留新记忆

1

vector recall

如果已有向量数据且 embedding 服务可用，batchDedup 用向量相似检索召回候选。

2

FTS recall

如果向量召回不可用但 FTS 关键词检索可用，则用关键词召回候选，去重判断的召回基础会变弱。

3

no recall

如果向量与 FTS 召回都不可用，跳过冲突检测，并把本批新记忆按默认 store 决策写入。

4

write path

writeMemory 按 store / update / merge / skip 决策维护 JSONL 与 store 写入；需要时可手动依据 JSONL 重建检索层。

为什么需要 L2 Scene BlocksWhy L2 scene blocks exist

L1 原子记忆适合搜索单个事实，但真实活动通常由许多小事实共同构成：一次调试、一次设计讨论、一次偏好变化。 L2 Scene Block 的作用，是把这些零散 L1 组织成可阅读、可导航、可追溯、可增长的情境块，再把稳定结论交给 L3 Persona。

🗂️ 生活类比
L1 像散落的便利贴：“喜欢中文回答”“今天在修 dedup”“证据来自消息 m42”。L2 像项目日志的一页： 把同一次活动的便利贴贴到一起，写上摘要、时间线、热度和证据链接。L3 则像长期画像，只保留多次日志反复支持的稳定特征。

L2 在 L1 与 L3 之间架桥

L1 atoms
可搜索的细粒度事实、偏好、事件、约束
->
L2 scene blocks
按活动聚合上下文、时间线、证据与摘要
->
L3 persona
从多个场景提炼稳定画像与长期倾向
->
recall
下一轮按需注入，不把每个原子都塞进 prompt

如果只用 L1，系统可以找到单条事实，却很难解释“这些事实属于哪一次活动、前后顺序是什么、哪些证据支持它、这件事最近是否还热”。 如果直接从 L1 生成 L3，又容易把一次性的噪声误当成长期人格。L2 因此是中间层：保留事件上下文与活动历史， 让 L3 只从经过场景聚合的证据里提炼稳定特征。

L1 与 L2 的职责边界

L1 atom responsibilities

原子化、可搜索、可去重。每条记录尽量表达一个事实，带类型、优先级、时间与 source_message_ids，方便 recall 和证据下钻。

L2 scene responsibilities

人类可读、情境化、可导航。一个 scene block 聚合多条相关 L1，维护活动摘要、时间线、来源引用、热度与更新历史。

Scene Block 的已解析 META 字段

字段作用
created记录场景首次形成的时间，用于区分老背景与新活动。
updated记录最后一次合并新 L1 的时间，形成活动时间线。
summary用少量文字描述这次活动的核心上下文，避免把所有 L1 原子直接注入。
heat表示近期活跃度或重要性，帮助后续 recall 与整理流程优先处理热场景。

这些是 parseSceneBlock 实际解析/格式化的 META key。证据不是一个被解析的 META 字段： 它应保留在 scene 正文叙事、来源引用和链接中，说明哪些 L1 原子支持这段摘要；需要核实时，再通过 L1 的 source_message_ids 回到 L0 原文。

这些字段和正文叙事共同解决一个核心问题：L2 保存“这批事实为什么有关联”。它不会替代 L1 搜索，也不会删除 L0 原文； 它把多条 L1 的关系、顺序与来源线索组织成可读块。这样 recall 可以先注入相关场景摘要，需要核实时再沿正文中的来源引用回到 L1 和 L0。

Scene index、导航与备份Scene index, navigation, and backups

上一课看到 LLM 只负责写 scene_blocks/。本课看工程代码如何在沙箱外维护 .metadata/scene_index.json，再生成可被 read_file 逐步下钻的 Scene Navigation。

🧭 生活类比
Scene block 是档案页，scene index 是管理员维护的目录卡，Scene Navigation 是贴在工作台上的索引摘要。 Agent 先看摘要判断哪页可能相关；需要证据时，再按绝对路径取出那一页，而不是把整柜档案每轮都搬进 prompt。

从 Markdown 场景到 recall 导航

scene markdown files
LLM 在 scene_blocks/ 写出的可读场景
->
parser
parseSceneBlock 解析 META
->
scene index JSON
工程侧写 .metadata/scene_index.json
->
navigation
按 heat 排序并带文件路径
->
recall
注入高层索引，按需 read_file

syncSceneIndex() 是沙箱边界之后的确定性整理步骤。它扫描 scene_blocks/*.md，对每个 Markdown scene 调用 parseSceneBlock 读取 created、updated、summary、 heat 等 META，并把 filename、summary、 heat、created、updated 写入 .metadata/scene_index.json。这个文件位于 LLM 工作区之外：LLM 可以改场景正文， 但不能伪造目录状态、覆盖 checkpoint，或绕过工程代码的清理与一致性检查。

Index file vs Navigation markdown

Index file

.metadata/scene_index.json 是机器可读状态：每条 SceneIndexEntry 只包含 filename、summary、heat、created、updated。它由 readSceneIndex 读取、由 writeSceneIndex 写入、由 syncSceneIndex 从真实 Markdown 重新生成。

Navigation markdown

generateSceneNavigation() 把索引变成 prompt 友好的 Markdown：热场景排在前面，显示摘要和安全的绝对路径示例，如 /workspace/agent-memory-data/scene_blocks/payment-debugging.md，方便 agent 用 read_file 逐步展开细节。

导航不是要替代场景全文，而是做 progressive disclosure。generateSceneNavigation() 只按 heat 降序排序，把热度最高的场景放在前面；每条导航保留文件路径，并显示更新时间， 让 agent 在判断“这条摘要相关”之后再读取完整 Markdown。这样 recall 可以花很少的 token 暴露场景地图， 同时仍保留回到完整 L2 叙事和来源引用的能力。

为什么追加到 persona 与 recall context

Scene Navigation 会被追加到 persona 和 recall 的系统上下文中，因为 L3 画像与本轮 recall 都需要知道“有哪些可用场景”。 Persona 负责稳定长期倾向，但场景导航告诉 agent：某个长期特征背后有哪些近期活动可查；recall 负责本轮相关性， 导航则让它先看摘要，再用 read_file 下钻完整证据。注入导航而非全文，可以同时保留可发现性与 prompt budget。

PersonaGenerator 与增量画像更新PersonaGenerator and incremental profile updates

PersonaGenerator 把 L2 scene blocks 提炼成 L3 persona.md：第一次生成建立稳定画像， 后续增量更新只优先阅读自上次 persona checkpoint 之后变化的场景，同时保留旧画像中的稳定信息。

🧑‍🏫 生活类比
L2 场景像一叠项目日志，L3 persona 像交给新同事的长期协作简介。第一次要读足够多日志形成简介； 之后不应每次重写整个人设，而是查看最近新增/改动的日志，在保留稳定结论的基础上修订简介。

L3 不是证据层，而是注入上下文

L1evidence atoms
细粒度事实与 source_message_ids，负责可搜索、可追溯。
L2scene blocks
把相关 L1 聚合成活动叙事、时间线、摘要和场景导航。
L3persona.md
稳定偏好、长期目标、协作方式与背景摘要，用于下一轮注入。
⚠️ 设计警告
persona 是被注入给 agent 的高层上下文，不是不可变的事实来源。遇到冲突或需要审计时，应沿 Scene Navigation 回到 L2 场景，再下钻到 L1/L0 证据，而不是把 persona.md 当成最终真相。

第一次生成 vs 增量更新

First generation

当没有已有 persona，prompt 输入主要是工程已预加载的可用场景内容、场景总数/变化场景数和生成规则；scene index 由工程代码读取并汇总成这些输入。模型需要从 L2 场景内容中总结长期偏好、工作方式、项目背景和稳定约束，写出一个可读的 persona.md。

Incremental update

当已有 persona，工程代码先用 stripSceneNavigation 去掉旧导航，再把现有画像与变化场景一起交给 prompt。模型应保留仍成立的稳定信息，只吸收新证据支持的变化。

增量模式的关键游标是 last_persona_time。PersonaGenerator 读取 .metadata/recall_checkpoint.json 后，用 scene index 里的 updated 过滤出“自上次 persona 生成后变化过”的场景。这样 L3 不需要每轮重扫所有场景，也不会因为一次局部活动把长期画像全部重写。

L2/L3 调度与触发规则L2/L3 scheduling and trigger rules

MemoryPipelineManager 不在每条消息后立刻跑完整 L1/L2/L3。它把工作分层调度：L1 用阈值、空闲和 shutdown flush 收集批次； L2 在 L1 成功后按 delay/min/max/active-window 触发场景整理；L3 则在 L2 完成后进入全局串行队列，并用 pending 标记合并重复触发。

⏱️ 生活类比
L1 像等你“说完一小段”才记笔记：说够阈值就立刻记，没说够就等空闲。L2 像项目复盘：新笔记来了可以把复盘提前， 但不能因为你一直聊天而无限推迟。L3 像全公司共享的人物画像，只允许一个编辑一次更新，避免多人同时改同一份 brief。

入口是 notifyConversation(sessionKey, messages)。它先把消息追加到 session buffer， 增加 conversation_count 并刷新 last_active_time。 如果计数达到 warm-up 或稳态阈值，就直接 enqueueL1；否则重置 L1 idle timer。 这让活跃对话按批处理，低频对话也能在空闲后被捕获。

L1 resettable timer vs L2 downward-only timer

L1 resettable idle

每次未达到阈值的新对话都会重新 schedule() idle timer：旧倒计时取消，新倒计时从当前时刻重新开始。它的目标是 debounce：等用户停下来，再把残余 buffer 交给 L1。

L2 downward-only

L1 成功后计算 max(now + delay, lastL2 + minInterval)，再用 tryAdvanceTo()。如果新时间更早，就提前；如果更晚，就保持原计划，避免活跃会话一直把 L2 往后推。

L2 还承担两个保护：L2 调度路径（包括成功完成以及 retry/skip 收口）会用 maxInterval 重新设定下一次巡检，保证活跃 session 即使没有新 L1 也会定期整理； 周期性的 maxInterval 定时器触发时检查 sessionActiveWindowHours，冷 session 不再继续轮询，等下一次 L1 成功再重新唤醒。 因此 downward-only 不是“越快越好”，而是在 delay-after-L1 的响应性、minInterval 的限速和 maxInterval 的兜底之间取平衡。

L3 为什么全局串行

L2 是按 session 整理场景，L3 persona 却汇总所有场景并写同一个高层画像。多个 session 同时完成 L2 时，如果并发生成 persona， 它们会读取相近但不同步的 scene/persona 状态，最后写入者可能覆盖先完成者。为避免这种全局写冲突， enqueueL3 使用一个 SerialQueue("L3")、一个 l3Running 和一个 l3Pending：运行中收到新触发只标 pending，当前 L3 结束后最多再补跑一次。

1

停止接收新工作

destroyed 先置为 true；从这一刻起，advanceL2Timer() 和 triggerL3() 都会直接返回，不再安排新的 L2/L3。

destroyed = true
2

flush L1 idle

取消 L1 idle timer；如果 session 还有 buffered messages，只在安全时用 enqueueL1(..., "flush") 收尾。

cancel idle -> optional L1 flush
3

等待已入队 L1 drain

等待 L1 queue 清空；L1 成功会先持久化 PipelineSessionState，但 shutdown 期间不会再推进新的 L2 timer。

persist state -> await l1Queue.onIdle()
4

flush L2 timer 后 drain L2/L3

_doFlush() 会先把 pending 的 L2 schedule timer 刷入 L2 queue，再等待 L2/L3 queue 清空；因为 destroyed 已为 true，这些 L2 run 成功后只持久化状态，不会再 arm 新 timer 或触发 L3。

flush L2 timers -> drain L2/L3
5

最终持久化状态供恢复

L1/L2 正常成功路径已经在调度前保存状态；destroy 末尾再保存一次，保留 pending 计数与时间戳，便于下次启动恢复未整理的工作。

persist state -> recover later

Prompt 构建前的 Auto RecallAuto-recall before prompt build

Auto Recall 发生在 prompt 真正交给模型之前。插件先保留用户原始输入，并把原始 event.prompt 交给核心入口； 真正进入 FTS/embedding 检索前，searchMemories 再调用 sanitizeText(userText) 清洗查询。 如果召回及时返回，就把用户侧上下文和系统侧上下文分别注入。如果超时或失败，主对话必须继续，不让记忆增强阻塞用户响应。

🧭 生活类比
这像开会前的助理检索：用户刚提出问题，助理快速翻出相关项目笔记，贴在发言稿前后。 如果档案柜一时打不开，会议不会停在门口；最多这轮先按没有记忆的方式回答。

1

用户文本进入 hook

index.ts 的 before-prompt hook 收到本轮用户文本，并把原始 prompt 写入 pendingOriginalPrompts，供后续 capture 使用。

user text -> pendingOriginalPrompts
2

交给 TdaiCore

api.on("before_prompt_build") 调用 TdaiCore.handleBeforeRecall 时传入的是原始 event.prompt，让宿主壳只负责接线，核心层负责 recall 语义。

index.ts -> TdaiCore.handleBeforeRecall(raw prompt)
3

执行超时竞速

performAutoRecall 使用 recall.timeoutMs 与召回任务 race：配置策略决定查哪些层、取多少上下文，超时则返回跳过注入。

recall task -> race(timeoutMs)
4

注入上下文

及时返回时，结果里的 prependContext 被放到用户消息前，appendSystemContext 被追加到系统上下文。

prependContext + appendSystemContext -> prompt
5

模型继续响应

模型看到的是增强后的 prompt；如果召回超时或失败，模型仍然收到原始对话路径，不被记忆系统拖住。

prompt -> model

两种注入位置

Prepend user context

适合放“与本轮用户问题直接相关”的记忆片段。它贴近用户文本，让模型把这些事实当成本轮问题的可用背景，而不是新的用户命令。

Append system context

适合放更稳定的系统侧说明，例如 recall 结果摘要、来源提示或使用边界。它补充系统上下文，但不覆盖原始系统规则。

缓存、清洗与指标

original prompt cache
pendingOriginalPrompts 保留用户原文，避免 capture 保存被 recall prepend 污染后的文本。
->
sanitized query
searchMemories 在 FTS/embedding 查找前调用 sanitizeText(userText)，防止把注入标签或控制片段带入检索。
->
recall result
performAutoRecall 生成可注入上下文、命中信息与耗时。
->
metrics cache
pendingRecallCache 按 sessionKey 暂存 recall payload，供 agent_end 上报 agent_turn 指标。

核心入口收到的是原始 prompt；清洗发生在 searchMemories 内部，清洗后的 query 用于“找记忆”，原始 prompt 用于“记录用户真的说了什么”。这两个数据面不能混淆： 如果 capture 直接保存被 prepend 后的文本，下一轮 L0/L1 会把系统注入当成用户证据，形成自我污染。

核心伪代码

before_prompt_build(userText):
    pendingOriginalPrompts[sessionKey] = userText
    result = await core.handleBeforeRecall(userText, sessionKey)
    if result within timeout:
        prepend_to_user(result.prependContext)
        append_to_system(result.appendSystemContext)
        pendingRecallCache[sessionKey] = result.recallPayload
    else:
        continue_without_memory()

L1 记忆搜索与召回预算L1 memory search and recall budgeting

第 22 课看到 Auto Recall 在 prompt 构建前启动；本课下钻到 L1 搜索本身。 searchMemories() 先清洗用户文本，再按 keyword、embedding 或 hybrid 选择检索路径。 召回的产物不是完整 transcript，而是足够短、足够相关的 L1 片段；更长的证据应通过显式 memory search 工具继续查。

🧭 生活类比
这像会议助理只把三五条最相关的便签贴到发言稿旁边，而不是把整本会议记录塞进来。 便签可以由云端档案柜一次完成“关键词 + 语义”混合查找，也可以由本地助理分别查两份清单后再合并；如果配置里根本没有语义设备，就按关键词查。

searchMemories 的召回管线

sanitized query
sanitizeText(userText) 去掉不可信控制片段，只保留适合检索的文本。
->
strategy
配置可选 keyword、embedding、hybrid；未显式配置时按 recall 配置默认值决定。
->
candidates
FTS5 BM25 和/或向量余弦相似度产生候选 L1。
->
threshold / hybrid rank
关键词或 embedding 单路结果用 scoreThreshold 过滤；混合路径优先由原生 store 执行，否则本地候选用 RRF 排名。
->
budget
maxResults 与字符预算限制注入规模。
->
injected lines
只注入 formatMemoryLine() 生成的简洁 L1 行：- [type|scene] content，可附活动时间；id 和 score 不进入 prompt，解析注入文本的指标不应依赖 score。

三种策略怎么选

keyword

searchByKeyword 走 SQLite FTS5。buildFtsQuery 把清洗后的词变成 FTS 查询，结果用 BM25 排序，适合精确术语、文件名、函数名。

embedding

searchByEmbedding 先把 query 编成向量，再用余弦相似度找语义接近的 L1。它适合用户换一种说法提问，但依赖 embedding 资源可用。

hybrid

混合召回有两条路：如果 store 声明 nativeHybridSearch，例如 TCVDB 的 hybridSearch，就让服务端完成 dense + sparse/RRF-like 混合；SQLite 或非原生路径才由本地 searchHybrid() 分别跑 keyword 与 embedding，再在 auto-recall.ts 内部做 RRF 合并。

降级规则要读窄一点：当配置选择 embedding 或 hybrid，但没有配置 vector store 或 embedding service 时， searchMemories() 会走 keyword 路径。 这不是“任何运行时向量/index 错误都必然降级”的承诺；实际异常仍受调用链、timeout 和错误处理保护，主对话应继续，但本轮不一定还能拿到关键词结果。

预算控制点

控制项作用为什么需要
maxResults限制最多格式化多少条 L1避免命中很多时把 prompt 挤满。
scoreThreshold过滤 keyword 或 embedding 单策略候选阈值主要作用在单路分数上；本地 hybrid 更依赖候选数量、RRF 排名、maxResults 和字符预算，原生 hybrid 则接收 topK/store-side 行为。
timeoutMs保护召回耗时搜索慢或资源异常时，主对话继续。
line truncation把每条 memory 行裁成短摘要L1 负责给模型线索，不负责搬运完整 transcript。
applyRecallBudget按总字符预算截断最终注入文本让 recall 有边界，给用户问题和系统规则保留空间。

L2 导航与 L3 Persona 注入L2 navigation plus L3 persona injection

第 22、23 课解释了 Auto Recall 与 L1 搜索。本课看召回结果如何分层注入：每轮变化的 L1 片段进入 prependContext，贴在 user prompt 前；相对稳定的 L3 persona、L2 Scene Navigation 与工具指南进入 appendSystemContext，追加到 system prompt 末尾。这样既让模型看到长期画像和场景地图，又尽量不破坏 provider 的 prompt cache。

🧭 生活类比
这像助理给你准备会议材料：今天临时相关的便签放在发言稿最前面；长期稳定的个人偏好、项目目录和“需要时去哪里查”的说明放在会议手册后附录。 便签每轮都换，附录大多不变，所以缓存友好。

为什么不是注入完整场景

自动注入

自动路径只提供地图：场景文件 path、热度、更新时间和 summary。它告诉模型“哪里可能有完整证据”，但不把所有 scene block 塞进 prompt。

显式工具搜索

当回答需要细节时，agent 应主动调用 read_file 读取相关场景，或用 tdai_memory_search 查 L1、tdai_conversation_search 查 L0 原文证据。

Scene Navigation 的设计是 progressive disclosure：先给索引，不给全集。完整场景可能很长，而且不是每轮都相关；把它们全部自动注入会挤占用户问题、系统规则和 L1 相关片段。 导航里的绝对路径让 agent 在确实需要“项目背景、事件经过、阶段结论”时，用 read_file 精准下钻。

🧰 MEMORY_TOOLS_GUIDE 运行时约束
  • tdai_memory_search 查 L1 structured memory：适合找结构化事实、偏好、决策与摘要。
  • tdai_conversation_search 查 L0 raw conversation evidence：适合回到原始对话证据做补充或核验。
  • read_file 用于 Scene Navigation 给出的路径：先看地图，再读取相关 scene 文件下钻。
  • tdai_memory_search + tdai_conversation_search 每轮总共最多 3 次；如果 3 次搜索仍未命中，停止继续搜索，基于已有上下文回答。

为什么更利于 prompt cache

很多 provider 的 prompt cache 对稳定前缀或稳定 system 区域更友好。L3 persona、L2 navigation 和工具指南变化频率低，放在 system 末尾可以在连续多轮中保持相同内容；L1 召回每轮跟着用户问题变化，如果也放进 system，就会频繁让系统上下文失效。 把动态 L1 移到 user 前缀，既保持本轮相关性，也让稳定 system-tail 内容更可能复用缓存。

SQLite、sqlite-vec、FTS、BM25 与混合搜索SQLite, sqlite-vec, FTS, BM25, and hybrid search

本课把第 23 课的策略落到本地 SQLite store。VectorStore 在同一个 vectors.db 里维护元数据表、FTS5 虚拟表和 sqlite-vec 的 vec0 向量表：关键词召回用 FTS5/BM25，语义召回用 cosine distance， 混合召回在客户端做 RRF-style merge：使用共享 helper 或调用路径中的本地等价实现合并排名。

🧭 生活类比
这像一个本地资料柜有三套索引：登记表记录每份材料的 id、时间和分类；关键词卡片按词找材料； 语义卡片按“意思相近”找材料。最后助理把两份候选清单做 RRF-style 融合，避免只相信一种索引。

本地 store 的四层结构

metadatal1_records / l0_conversations
普通 SQLite 表保存可展示、可过滤、可追溯的字段：L1 content/type/priority/scene/timestamps，L0 session、role、message_text 与 recorded_at。
keywordl1_fts / l0_fts
FTS5 虚拟表保存分词后的索引列，并保留 original 文本用于展示；查询由 buildFtsQuery() 构造，结果按 bm25() rank 升序排列。
vectorl1_vec / l0_vec
vec0 表保存 embedding，使用 distance_metric=cosine 做近邻搜索；维度来自 embedding 配置。
embeddingprovider + model + dimensions
embedding service 记录 provider/model/dimensions。provider、model 或维度变化会让旧向量不可比；store 初始化结合 embedding metadata 判断并触发 drop vec tables + reindex 路径。

写入路径：一个事务包住元数据、FTS 和向量

write L1
L1 writer 传入 record 与可选 embedding。
->
metadata insert
l1_records 用 ON CONFLICT 更新普通字段。
->
FTS tokenize
tokenizeForFts() 写 l1_fts；FTS 失败只降级关键词索引。
->
embedding
有可用向量时进入 vec0；无 embedding 时仍保留 metadata + FTS。
->
vector upsert
vec0 不支持 ON CONFLICT，所以先 delete 再 insert。

sqlite.ts 使用手动 BEGIN/COMMIT， 失败时 ROLLBACK，让 metadata、FTS 和 vector 尽量保持同一条记录的同步状态。 数据库开启 WAL mode，读写可以更稳地并行；但要记住，FTS 与 vec0 是派生索引，metadata 表才是可追溯事实的主记录。

三种召回列对比

BM25 keyword recall

buildFtsQuery(query) 把用户输入变成可执行的 FTS5 查询；bm25(l1_fts) 或 bm25(l0_fts) 排名。适合精确术语、函数名、文件名、中文分词后关键词。

Vector semantic recall

查询先经过同一个 embedding provider 编码，再在 l1_vec/l0_vec 上按 cosine distance 取最近邻。适合“换一种说法”的语义相似。

RRF-style hybrid

本地 SQLite 没有服务端原生 hybrid rank；工具层分别拿 FTS 和 vector 候选，再用共享 RRF helper 或调用路径中的本地等价实现按名次加权融合，得到更稳的候选顺序。

核心伪代码

writeMemory(record):
    begin_transaction()
    upsert(l1_records, record.metadata)
    upsert(l1_fts, tokenize_for_fts(record.content))
    if embedding_available:
        delete(l1_vec where record_id = record.id)
        insert(l1_vec, embed(record.content))
    commit()

hybrid(query):
    ftsRanked = searchL1Fts(buildFtsQuery(query))
    vecRanked = searchL1Vector(embed(query))
    return rrf_style_merge_or_local_equivalent([ftsRanked, vecRanked])

provider metadata 与 reindex

向量不是“纯文本索引”，它绑定了 provider、model 和 dimensions。同一段文字由不同模型生成的向量空间可能完全不同； 维度不同则 vec0 表结构也不兼容。因此 embedding_meta 检测到 provider/model/dimensions 改变时，要保留 l1_records 与 l0_conversations，重建 l1_vec/l0_vec， 再按当前 embedding service 重新索引。local provider 可能需要 warmup；remote provider 是 HTTP 调用，但二者都必须产出与配置维度一致的向量。

本地 SQLite 与云端 TCVDB 对比

SQLite local backend

元数据、FTS5、sqlite-vec 和 JSONL 都在本地。混合搜索通常由客户端分别查 keyword/vector，再做 RRF-style 合并。优点是零云依赖、便于离线；限制是索引能力和并发能力跟随本机环境。

Tencent Cloud VectorDB backend

L1/L0 集合、dense embedding 字段、sparse vector、filter indexes 和 native hybridSearch 下沉到 VectorDB；profiles 集合仅做 metadata/profile 同步查询。优点是统一远程检索、服务端 RRF rerank 与迁移/导出路径；要求配置、网络和凭据都可用。

为什么长任务日志需要符号压缩Why long task logs need symbolic compression

长任务里的工具日志会迅速膨胀：同一个文件被反复读取，命令输出一屏接一屏，早期结果很快变旧却仍占着 live context。 Context Offload 不是替代 L0-L3 长期记忆，而是在当前任务内把这些短期证据压成可导航符号：原始输出落到 refs 文件，紧凑行写入 offload JSONL， 任务画布渲染成 Mermaid MMD，下一轮只把继续工作所需的符号注入回上下文。

🧭 生活类比
L0-L3 像跨项目档案馆，帮未来会话记住事实、场景和画像；Context Offload 像桌面上的任务白板。 白板不保存一切细节，只把“证据在哪里、当前做到哪一步、下一步该看哪个节点”保持在眼前。

为什么长任务日志会挤爆窗口

重复工具调用

编码会话常反复 rg、read file、run test。每次结果都可能被追加进消息历史，即使后续只需要“哪个文件已确认”这个符号。

大块命令输出

构建、测试和日志可能产生成百上千行。原始输出是证据，但下一轮推理通常只需要失败摘要、引用路径和当前状态。

过期结果滞留

早期搜索结果、旧失败栈和已修复错误会继续消耗 token。它们应可追溯，但不该一直挤占实时 prompt window。

长期 L0-L3 与短期 Context Offload

长期 L0-L3 memory

面向未来会话：L0 留原始对话证据，L1 抽取原子记忆，L2 组织场景块，L3 生成 persona/profile。目标是跨会话复用、搜索和长期演化。

短期 Context Offload

面向当前任务：把工具结果从 live context 移到本地证据文件和任务符号层。目标是让长编码会话可继续、可下钻、可控 token 预算。

共同点与边界

两者都强调证据路径和压缩摘要；差别是生命周期。Offload 解决当前任务导航，不承诺成为跨 session 的长期事实来源。

1

解析工具事件

createAfterToolCallHandler 从事件中解析 tool call ID、工具名、参数和结果；如果事件缺少可用 ID，后续 dedup 就无法稳定工作。

event -> tool_call_id + params + result
2

过滤非结果事件

heartbeat 只说明宿主还活着；approval-pending 只说明工具尚未真正执行。两者都不应进入证据层，避免把“尚无结果”当成事实。

3

stateManager.addToolPair 只负责缓存已经完成的工具调用；latest-turn context cache 是 after-tool-call 中另一条独立分支，用来保留本轮任务框架。

4

shouldForceL1 使用 forceTriggerThreshold 判断 pending pairs 是否足够多；达到阈值就走 L1 trigger orchestration。

pending_pairs >= forceTriggerThreshold

L1 输出变成 OffloadEntry；原始结果由 writeRefMd 写入 refs，紧凑行由 appendOffloadEntries 追加。

Mermaid MMD 画布、node_id 与下钻恢复Mermaid MMD canvas, node_id, and drill-down recovery

Mermaid MMD 不是为了好看，而是 Context Offload 的紧凑任务状态画布。Offload-L2 可以完整写入新的 .mmd 文件，也可以按行修补已有画布；但无论是新节点还是补丁里的新工具条目，都必须拿到 node_id。这个 ID 把注入到 prompt 的任务符号，重新连接回 offload JSONL 摘要行和 refs/*.md 原始证据。

🧭 生活类比
把 MMD 当作项目作战地图：地图上一个节点写着“修复登录测试”。node_id 像坐标编号， 让你从地图坐标回到索引卡，再翻到证据夹里的原始日志。没有坐标，地图再漂亮也无法复盘证据。

MMD 是任务状态画布，不是装饰图

canvasmmds/task.mmd
用 Mermaid 节点表达当前任务、等待项、已处理项和后续路径；L2 负责 writeMmd 完整写入或 patchMmd 行级修补。
indexoffload JSONL rows
每行保存 tool_call_id、summary、result_ref、offloaded 和 node_id；updateOffloadNodeIds 把映射写回行。
evidencerefs/*.md
完整工具输出、错误栈或搜索结果留在 refs markdown；prompt 里不需要常驻所有原始行。

因此评审 MMD 生成时，不应只问“图是否能渲染”。更重要的问题是：每个新工具证据是否被放到明确节点？ 节点 ID 是否符合 MMD_NODE_ID_RE？映射是否能被规范化后回填到 JSONL？ 如果任何一步断开，Offload-L3 注入的 active node 就只剩摘要，无法安全下钻。

node_id 生命周期

状态含义恢复能力
nullL1 刚写入的未分配行；还没有进入 MMD 画布。能按时间或 tool_call_id 找到摘要和 ref，但不能从 MMD 节点直接定位。
wait等待态或兜底映射；L2 尚未能归入稳定任务节点。可保守保留，不应伪造具体节点；后续 L2 可再归一化。
027-N1具体节点 ID，符合 MMD 节点命名规则并出现在 node_mapping。MMD node -> JSONL row -> refs detail 的主下钻路径完整。
deleted / offloaded行可能被标记为已 offload 或删除态；状态不等于丢失证据。注入层可跳过或压缩显示，但仍应能通过索引和 ref 追溯历史。
