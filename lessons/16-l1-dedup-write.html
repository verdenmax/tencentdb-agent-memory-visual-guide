第四部分 · L0 与 L1：捕获、抽取、去重Part 4 · L0 and L1: capture, extraction, dedup

L1 去重、冲突检测与写入L1 dedup, conflict detection, and write path

L1 抽取会把多条 L0 消息变成结构化记忆，但抽取结果可能重复，也可能与已有记忆冲突。写入前的 batchDedup 是质量控制：先召回相似候选，再判断这条新记忆应该保留、更新已有记忆，还是跳过。 它控制的是可搜索 L1 事实的质量，而不是删除 L0 原文证据。

🧩 生活类比
L0 像会议录音库，L1 像团队知识卡片。录音库不能因为有人做了新卡片就剪掉原声； 去重只是检查“这张卡是不是已经有了、是不是需要修订旧卡、是不是与旧卡冲突”，让知识卡片架保持清晰。

从抽取结果到写入决策

extracted memories
LLM / logic 产出候选 L1 原子记忆
->
candidate recall
向量 store 可用时召回相似已有记忆
->
dedup decision
判断 keep / update / skip 与冲突说明
->
write / update / skip
只把合格 L1 进入 JSONL 与可搜索 store

src/core/record/l1-dedup.ts 中的 batchDedup 会在批量写入前处理抽取结果。它优先利用 store 的相似检索找候选：同一偏好被重复表达时可跳过； 用户纠正旧偏好时可更新；新事实没有相似候选时保留。判断可以来自 src/core/prompts/l1-dedup.ts 中的 LLM prompt，也可以在降级路径中使用规则逻辑。

为什么去重不是丢弃证据

L0 证据层

保存原始对话 JSONL：谁说了什么、什么时候说、消息 ID 是什么。它服务审计、回放和重新抽取，去重不应改写这层原始证据。

L1 质量层

保存可召回的结构化事实。去重只决定新 L1 是否会让事实库重复、过时或冲突，从而控制未来 recall 的噪声。

写入路径：JSONL 与 store upsert

通过去重的记忆进入 writeMemory。writer 先用 generateMemoryId 生成稳定记录 ID，再把 L1 记录追加到 JSONL。 随后 store upsert 路径把同一条记录维护到可搜索后端：本地 sqlite 可提供向量 / FTS 检索，远端 TCVDB 可提供托管向量检索。JSONL 是持久证据，store 是检索索引；两者一起让记忆既可审计又可召回。

SQLite local store

src/core/store/sqlite.ts 适合零配置或本地开发。它在本地数据库维护向量与 FTS 索引，让 L1 记录即使没有远端服务也能被相似搜索或关键词搜索找到。

TCVDB backend

src/core/store/tcvdb.ts 适合接入托管向量数据库。createStoreBundle 根据配置组装 embedding、store 与 fallback，使上层写入逻辑不需要知道具体后端。

降级模式：embedding / store 不可用时仍保留记忆

1

embedding unavailable

无法生成向量时，相似召回质量下降，batchDedup 只能使用已有候选、关键词或保守规则。

2

store unavailable

无法 upsert 索引时，系统不应把 L1 结果直接丢弃；索引只是检索加速层。

3

JSONL fallback

writeMemory 仍追加 L1 JSONL，保留内容、来源消息、类型、优先级和时间元数据。

4

later rebuild

服务恢复后，可依据 JSONL 重新建立 store 索引；证据链没有因为临时故障丢失。

伪代码

for memory in extracted:
    candidates = recall_similar(memory)
    decision = dedup(memory, candidates)
    if decision.keep:
        record = writeMemory(memory)
        store.upsert(record)
    elif decision.update:
        update_existing(record)

  • src/core/prompts/l1-dedup.ts：去重 prompt 描述如何比较新记忆与候选记忆、发现重复或冲突
  • src/core/record/l1-writer.ts：writeMemory 与 generateMemoryId 负责生成并追加 L1 JSONL 记录

✅ 本课要点
L1 去重发生在写入前，因为抽取可能产生重复或冲突记忆；batchDedup 在 store 可用时召回相似候选并决定 keep / update / skip；writeMemory 写入 L1 JSONL，store upsert 维护可搜索索引；embedding 或 store 不可用时，JSONL fallback 仍保留可审计的记忆证据。

L1 extraction turns L0 messages into structured memories, but the extracted batch can contain duplicates or facts that conflict with existing memory. Before writing, batchDedup acts as quality control: recall similar candidates, decide whether the new memory should be kept, used to update an existing memory, or skipped. It governs searchable L1 facts; it does not delete raw L0 evidence.

🧩 Analogy
L0 is the meeting recording archive; L1 is the team's index-card shelf. You do not cut the recording when a new card is written. Dedup only asks whether the card already exists, should revise an older card, or conflicts with one, so the shelf stays useful.

From extraction to write decision

extracted memories
LLM / logic emits candidate L1 atoms
->
candidate recall
when vector store is available, recall similar existing memories
->
dedup decision
choose keep / update / skip and explain conflicts
->
write / update / skip
only qualified L1 enters JSONL and searchable store

batchDedup in src/core/record/l1-dedup.ts runs before batch writes. It prefers store similarity search for candidates: repeated preferences can be skipped, corrected preferences can update an existing record, and unrelated new facts can be kept. The decision can come from the LLM prompt in src/core/prompts/l1-dedup.ts, or from conservative rule logic in degraded paths.

Why dedup does not discard evidence

L0 evidence layer

Raw conversation JSONL records who said what, when, and under which message ID. It supports audit, replay, and re-extraction; dedup should not rewrite this source evidence.

L1 quality layer

Structured facts are optimized for future recall. Dedup only decides whether a new L1 fact would make the fact store repetitive, stale, or conflicting.

Write path: JSONL plus store upsert

Accepted memories enter writeMemory. The writer uses generateMemoryId to create a stable record ID, then appends the L1 record to JSONL. After that, the store upsert path keeps the same record searchable: local sqlite can provide vector / FTS search, while TCVDB can provide managed vector search. JSONL is durable evidence; the store is the retrieval index. Together they make memory auditable and recallable.

SQLite local store

src/core/store/sqlite.ts fits zero-config or local development. It maintains local vector and FTS indexes so L1 records remain discoverable through similarity or keyword search without a remote service.

TCVDB backend

src/core/store/tcvdb.ts integrates a managed vector database. createStoreBundle wires embedding, store backend, and fallback behavior so the writer does not depend on backend details.

Degraded mode: keep memory when embedding / store is unavailable

1

embedding unavailable

Without vectors, similar-candidate recall becomes weaker, so batchDedup must use available candidates, keywords, or conservative rules.

2

store unavailable

If index upsert fails, the system should not drop L1 output; the store is a retrieval accelerator, not the only persistence path.

3

JSONL fallback

writeMemory still appends L1 JSONL with content, source messages, type, priority, and time metadata.

4

later rebuild

When services recover, the store index can be rebuilt from JSONL; the evidence chain was not lost during the outage.

for memory in extracted:

    decision = dedup(memory, candidates)
    if decision.keep:

✅ Key points
L1 dedup runs before writing because extraction can produce duplicate or conflicting memories. batchDedup recalls similar candidates when the store is available and chooses keep / update / skip. writeMemory writes L1 JSONL, store upsert maintains the searchable index, and JSONL fallback still preserves auditable memory evidence when embedding or store services are unavailable.
