L3 上下文注入与紧急压缩L3 context injection and emergency compression

L3 是 Context Offload 真正进入模型前的最后一道运行时循环：先在 prompt 构建前注入当前 active MMD， 再把已经确认 offload 的旧工具区域按确定性快路径替换掉，随后用精确 token 快照判断压缩档位：中等压力只跑温和替换， 达到更高激进阈值时先激进删除，再做温和清理，最后才按需紧急压缩。目标不是“尽量删”，而是在不破坏当前任务、最新用户消息和工具调用结构的前提下，把模型输入压回安全窗口。

🧭 生活类比
像飞机起飞前的配载检查：任务地图先放进驾驶舱；已托运的旧行李用行李票替代；中等超重时先换成更轻的行李票， 严重超重时先把旧箱子移到货舱清单，再清掉最容易替代的包，最后才启动紧急减载。乘客、驾驶指令和安全联络不能被扔掉。

最后一圈发生在模型调用前

1

before prompt

before-prompt-build.ts 跳过内部 memory-pipeline session，读取 active MMD，并准备已确认 offload 的快路径。

skip internal pipeline; load active MMD
2

inject active MMD

injectMmdIntoMessages 用 marker 管理注入片段，避免重复追加；任务地图优先进入上下文。

MMD_MESSAGE_MARKER -> task map
3

token snapshot

context-token-tracker.ts 与 l3-token-counter.ts 计算当前消息的 token 使用，而不是凭字符长度猜测。

messages -> precise token count
4

tier gate

精确计数低于温和阈值则不动；50-85 一类中等压力只跑 compressByScoreCascade，达到更高激进阈值时先跑激进删除。

mild threshold < aggressive threshold
5

aggressive -> mild -> emergency

aggressiveCompressUntilBelowThreshold 先删除旧 offloaded 区域并注入 history MMD；随后温和清理最高可替代分区域，仍超紧急阈值才进入 emergencyCompress。

history MMD -> high-score replacement -> emergency trim

三档压缩策略

激进：删除旧 offload 区域

触发：token 达到高于温和线的激进阈值。
动作：aggressiveCompressUntilBelowThreshold 先删除更老的已 offload 消息块，并补入 history MMD。
恢复证据：从历史 MMD 节点回 JSONL，再回 refs。

温和：score cascade

触发：token 达到温和阈值；若先跑过激进删除，也继续作为清理步骤。
动作：compressByScoreCascade 按可替代分降序替换旧工具结果，高分先替换，低分/难总结证据保留更久。
恢复证据：保留 tool_call_id、node_id、result_ref 与摘要。

紧急：overflow 兜底

触发：激进和温和清理后 token 压力仍高，或上一轮入口溢出设置了 _forceEmergencyNext。
动作：emergencyCompress 只保留安全骨架、最新用户消息、有效工具对、MMD task-map 消息和最小保留集。
恢复证据：依赖 active/history MMD 与 JSONL 状态标记。

快路径为什么要标记 JSONL 状态

状态含义下一轮行为
offloaded: truemarkOffloadStatus 记录这段工具结果已经被安全替换为符号摘要。before-prompt-build 可确定性重放替换，不需要重新推断是否压缩。
offloaded: "deleted"激进压缩已把旧的 live 区域移出上下文，但 JSONL 行和 refs 证据仍可追溯。L3 注入层跳过原始区域，改用 history MMD 和节点索引恢复语义。
node_id + result_ref把 MMD 节点、JSONL 摘要和 refs 原文连接起来。即使内容被删除，也能从任务地图下钻。

一次 LLM 调用的 hook 时间线

before_prompt_build
跳过内部 memory-pipeline session；注入 active MMD；重放 offloaded: true 的确定性替换；写 token guard 快照。
llm_input_l3
createLlmInputL3Handler 用精确计数按真实顺序执行：超过激进阈值先 aggressiveCompressUntilBelowThreshold，再用 compressByScoreCascade 清理；中等压力只跑温和替换；最后才 emergencyCompress。
model call
如果入口抛出 token overflow，isTokenOverflowError 设置 _forceEmergencyNext，让下一次越过 below-mild gate 的 L3 pass 走紧急压缩，而不是在错误处理里直接内联返回。
after_tool_call
工具返回后刷新 active MMD，报告 patch effectiveness，并让下一轮知道哪些替换真正节省了 token。

安全规则：压缩不能破坏对话结构

⚠️ Safety rules
  • 保留最新用户消息：它定义当前意图，不能为了省 token 删除。
  • 保持 tool-use / tool-result 成对有效：不能只删 tool-use 或只删 tool-result，避免宿主协议损坏。
  • 温和阶段保护当前任务节点：compressByScoreCascade / l3-helpers.ts 使用 currentTaskNodeIds，避免把 active MMD 指向的证据刚好替换掉。
  • 激进/紧急阶段保护结构边界：激进删除按旧 offloaded 区域前缀推进；紧急保留 MMD task-map 消息、最新用户消息、有效工具对和最小保留集，而不是继续做节点级 current-task 过滤。
  • JSONL 写清状态：使用 markOffloadStatus 标记 true 和 "deleted"，让快路径可重复。
  • 跳过内部 memory-pipeline session：记忆流水线自己的压缩/生成不应递归触发 Context Offload。

核心伪代码

on_before_prompt_build(messages, session):
    if is_internal_memory_pipeline_session(session):
        return messages
    active_mmd = read_active_mmd(session)
    messages = injectMmdIntoMessages(messages, active_mmd)
    messages = reapply_confirmed_offloads(messages, statuses=[true, "deleted"])
    snapshot = build_precise_token_snapshot(messages)
    return messages, snapshot

on_llm_input_l3(messages, snapshot, state):
    if snapshot.total_tokens < mild_threshold:
        return messages

    force_emergency = state._forceEmergencyNext
    state._forceEmergencyNext = False

    if snapshot.total_tokens >= aggressive_threshold:
        messages = aggressiveCompressUntilBelowThreshold(messages, inject_history_mmd=True)
        if deleted_entries:
            markOffloadStatus(deleted_entries, "deleted")

    if count_tokens(messages) >= mild_threshold:
        messages = compressByScoreCascade(messages,
            sort_replaceability_score="desc",
            protect_latest_user=True)
        if replaced_entries:
            markOffloadStatus(replaced_entries, true)

    if count_tokens(messages) >= emergency_threshold or force_emergency:
        return emergencyCompress(messages,
            preserve_latest_user=True,
            preserve_valid_tool_pairs=True,
            preserve_mmd_task_map_messages=True,
            preserve_min_keep=True)
    return messages

# inside on_llm_input_l3 outer try/catch:
except error:
    log_l3_error(error)
    if isTokenOverflowError(error):
        state._forceEmergencyNext = True  # consumed by a later L3 pass after the below-mild gate
    return messages  # swallow: llm_input_l3 logs and does not re-raise

L3 is the last runtime loop before Context Offload enters the model call. It first injects the current active MMD before prompt build, then deterministically re-applies replacements for already confirmed offloads, then uses precise token snapshots to choose the tier: medium pressure runs only mild replacement, while pressure at the higher aggressive threshold runs aggressive deletion first, then mild cleanup, and only then emergency compression if needed. The goal is not “delete as much as possible”; it is to return model input to a safe window without breaking the current task, the latest user message, or tool-call structure.

🧭 Analogy
Think of an aircraft weight check before takeoff: the task map goes into the cockpit first; old checked luggage is represented by claim tickets; medium overweight cases use lighter claim tickets, while severe overweight cases first move older boxes to the cargo manifest, then clear the easiest-to-represent bags, and only then start emergency shedding. Passengers, flight instructions, and safety communications cannot be thrown away.

before prompt

Precise counts below the mild threshold do nothing; medium pressure such as 50-85 runs only compressByScoreCascade, while the higher aggressive threshold runs deletion first.

mild threshold < aggressive threshold
5

aggressive -> mild -> emergency

aggressiveCompressUntilBelowThreshold first deletes old offloaded regions and injects history MMD; mild cleanup then replaces the highest replaceability-score regions, and emergencyCompress runs only if still above the emergency threshold.

history MMD -> high-score replacement -> emergency trim

Three compression tiers

Aggressive: delete old offloaded regions

Trigger: token usage reaches the aggressive threshold, which is higher than the mild threshold.
Action: aggressiveCompressUntilBelowThreshold first removes older offloaded message blocks and injects history MMD.
Evidence recovery: go from history MMD nodes to JSONL, then to refs.

Mild: score cascade

Trigger: token usage reaches the mild threshold; after aggressive deletion, it also runs as the cleanup step.
Action: compressByScoreCascade replaces old tool results by descending replaceability score, so high-score regions go first and lower-replaceability / hard-to-summarize evidence survives longer.
Evidence recovery: keep tool_call_id, node_id, result_ref, and summary.

Emergency: overflow fallback

Trigger: token pressure remains high after aggressive plus mild cleanup, or the previous entry overflow set _forceEmergencyNext.
Action: emergencyCompress keeps only the safe skeleton, latest user message, valid tool pairs, MMD task-map messages, and the minimum keep set.
Evidence recovery: rely on active/history MMD plus JSONL status markers.

Why the fast path marks JSONL status

StatusMeaningNext-turn behavior
offloaded: truemarkOffloadStatus records that this tool result was safely replaced by a symbolic summary.before-prompt-build can deterministically replay the replacement without re-guessing compression.
offloaded: "deleted"Aggressive compression moved an old live region out of context, while the JSONL row and refs evidence remain traceable.The L3 injection layer skips the raw region and recovers meaning through history MMD and node indexes.
node_id + result_refConnects the MMD node, JSONL summary, and original refs text.Even deleted content can be drilled down from the task map.

Timeline of one LLM call

Skip internal memory-pipeline sessions; inject active MMD; replay deterministic replacements for offloaded: true; write the token guard snapshot.
llm_input_l3
createLlmInputL3Handler uses precise counts in the real order: above the aggressive threshold it runs aggressiveCompressUntilBelowThreshold first, then compressByScoreCascade; medium pressure runs only mild replacement; emergencyCompress is last.
model call
If the entry throws token overflow, isTokenOverflowError sets _forceEmergencyNext so the next L3 pass that proceeds past the below-mild gate runs emergency compression, rather than returning inline emergency output from the error handler.
after_tool_call
After a tool returns, active MMD is refreshed, patch effectiveness is reported, and the next turn knows which replacements actually saved tokens.

Safety rules: compression must not corrupt conversation structure

⚠️ Safety rules
  • Preserve the latest user message: it defines the current intent and must not be deleted to save tokens.
  • Keep tool-use / tool-result pairs valid: never delete only one side of a pair, or the host protocol can break.
  • Protect current-task nodes during the mild stage: compressByScoreCascade / l3-helpers.ts use currentTaskNodeIds to avoid replacing evidence targeted by the active MMD.
  • Protect structural boundaries during aggressive/emergency stages: aggressive deletion advances by old offloaded-region prefixes; emergency keeps MMD task-map messages, the latest user message, valid tool pairs, and the minimum keep set rather than continuing node-level current-task filtering.
  • Write JSONL status clearly: use markOffloadStatus with true and "deleted" so the fast path is repeatable.
  • Skip internal memory-pipeline sessions: the memory pipeline's own compression/generation should not recursively trigger Context Offload.

✅ Key points
The correct L3 order is: inject active MMD and replay confirmed offloads first, then use precise token counts to choose the compression tier. The aggressive threshold is higher than the mild threshold: medium pressure runs only mild replacement, while high pressure runs aggressive deletion before mild cleanup; mild replacement sorts replaceability score descending, so high-score regions go first. Safety invariants still preserve latest-user and valid tool-pair structure. markOffloadStatus values true / "deleted" make deterministic fast-path replay possible on the next turn.
